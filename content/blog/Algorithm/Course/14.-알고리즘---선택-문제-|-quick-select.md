# 0. 선택 문제

이전 수업에서는 특별한 k에 대한 선택 문제 알고리즘을 살펴봤다.

> 최소한의 비교로 최소값(k = 1), 최대값(k = n), 최대값과 최소값, 가장 작거나 큰 두 수 찾기

이번 수업에서는 일반적인 k에 대한 선택 문제에 대해서 살펴볼 것이다.

> n개의 값과 k(1 <= k <= n) 라는 특별한 값이 입력으로 주어졌을 때, k번째로 작은 수 찾기

<br>

선택 문제에서 k = n / 2 이면, (n / 2) 번째로 작은 수를 찾아야 한다.

- 이는 오름차순 또는 내림차순으로 나열된 n개의 값의 중간에 있는 값이다.
   - 이러한 값을 '중앙값(Median)' 이라고 한다.
- 이렇게, 임의로 주어지는 k에 대해, k번째로 작은 수를 찾아야 하는 문제다.
   - 그리고, 비교 횟수를 최대한 적게 만들어야 한다.

<br>

선택 문제에는 두 가지 중요한 알고리즘 'Quick Select' 와 'Median of Medians(MoM)' 가 있다.

- 이번 수업에서는 그중에 하나인 'Quick Select' 라는 알고리즘에 대해 살펴볼 것이다.
- 다음 수업에서는 조금 더 복잡한 'Median of Medians' 알고리즘을 살펴볼 것이다.

# 1. Quick Select

우선, n개의 숫자가 들어있는 L이라는 리스트가 있다고 가정해보자.

```
L = [                         ]
```

## 1-1. 피벗 선택

리스트 L에 있는 n개의 숫자 중, 기준이 될 숫자(p) 를 선택한다.

```
L = [         p               ]
```

- 이 때, p는 'pivot' 이라고 부르며, 어떤 방법으로 선택해도 상관없다.
- 첫 번째 값(L[0]), 마지막 값(L[n - 1]), 또는 임의로(random) 선택한다.

## 1-2. 크기 분류

p와 다른 값을 비교하여, p보다 작은 값들과 큰 값들로 나눈다.

```
A = { p보다 작은 값 }
B = { p보다 큰 값 }
M = { p와 같은 값 }

L = [         p               ]

    [     ] [ p       ] [     ]
    └─ A ─┘ └─── M ───┘ └─ B ─┘
```

- p보다 작은 값들을 모아서 A, 큰 값들을 모아서 B, 같은 값들은 M에 집어넣는다.
- p와 다른 (n - 1) 개의 값들을 비교하기 때문에, 총 (n - 1) 번의 비교가 필요하다.
- 이 때, A, B, M 에 각각 몇 개의 값이 포함될 것인지는 알 수 없다.
   - 어떤 집합에는 값이 하나도 없을 수도 있고, 한 곳에 다 모여있을 수도 있다.
   - 값의 크기대로 나열하면 A, M, B 이며, 이 때, M에는 무조건 pivot 이 들어있다.
- 중요한 것은, A 또는 B에 있는 값들 사이의 대소 관계에 대해서는 모른다는 것이다.
   - A에 있는 값들은 pivot 보다 작고, B에 있는 값들은 크다는 것만 알 수 있다.

## 1-3. k의 위치 찾기

순서대로 나열된 집합 중, k번째로 작은 값이 어디에 있는지를 알아내야 한다.

### 1-3-1. |A| > k 의 경우

```
[     ] [ p       ] [     ]
└─ A ─┘ └─── M ───┘ └─ B ─┘
─-> k

if |A| > k:
    A에서 k번째 작은 값 ? 재귀
    => M ∪ B 에는 없음!
```

만약 A에 포함된 값의 개수가 k보다 크다면, k는 A에 포함된다.

- A에 들어있는 값들은 pivot 보다 작은 값들이며, 최소값과 두 번째로 작은 값도 포함된다.
- A에 들어있는 값의 개수가 k보다 크다면, k번째로 작은 값은 당연히 A에 들어있을 것이다.
- 이 때, 'A에서 k번째로 작은 값' 은 우리가 찾고 있는 '전체에서 k번째로 작은 수' 와 같다.
   - 또한, 이는 전체에서 k번째로 작은 값이 M과 B(M ∪ B) 에는 없다는 것을 의미한다.
- 따라서, M과 B에 있는 값들을 확인할 필요 없이, A에서 k번째로 작은 값만 찾으면 된다.

<br>

이 때, A에서 k번째로 작은 수는 재귀적으로 찾을 수 있다.

- L이라는 리스트에서 k번째로 작은 값을 찾기 위해서 Quick Select 를 적용했다.
- 다시 한 번, Quick Select 를 적용하여, A에서 k번째로 작은 숫자를 찾으면 된다.
- n개의 값이 들어있는 L이 아닌, 그보다 작은 A에 대해 같은 일을 반복하는 것이다.

<br>

> 이렇게, L에서 k번째로 작은 수를 찾는 문제가 A에서 k번째로 작은 수를 찾는 문제로 바뀌었다.

### 1-3-2. |A| + |M| < k 의 경우

```
[     ] [ p       ] [     ]
└─ A ─┘ └─── M ───┘ └─ B ─┘
   └────┬────┘
   (|A| + |M|) < k
|─────────────────|─-> k
                    ↑
             (k - |A| - |M|)

elif |A| + |M| < k:
    B에서 찾아야 함
         ↖ (k - |A| - |M|) 번째 작은 값
```

A와 M에 들어있는 값의 개수를 다 합쳐도, k보다 작은 경우도 있다.

- 이 때, k번째로 작은 값은 A와 M이 아닌, B에 있는 값 중 하나가 될 것이다.
- k번째로 작은 값은 A와 M에 해당하는 범위를 지나치게 되기 때문이다.
- 따라서, 전체에서 k번째로 작은 수는 A와 M이 아닌, B에서 찾아야 한다.

<br>

여기서 중요한 점은, 'B에서 k번째로 작은 값' 을 찾아서는 안 된다는 것이다.

- 왜냐하면, B가 아닌 '전체에서 k번째로 작은 수' 를 찾아야 하기 때문이다.
- 이를 위해, 우선, k에서 'A와 M에 들어있는 값의 개수' 를 뺀 값을 구해야 한다.
- 따라서, 이 경우에는 'B에서 (k - |A| - |M|) 번째로 작은 값' 를 찾아야 한다.

### 1-3-3. 그 이외의 경우

```
else: # M
    return p
```

- k번째로 작은 수가 A에도 없고, B에도 없다는 것은, M에 있다는 것을 의미한다.
- M에 있는 값들은 모두 pivot 과 같으므로, k번째로 작은 값은 pivot 이 된다.

<br>

> 따라서, pivot 의 값을 그대로 반환하면 된다.

## 1-4. 의사 코드 작성하기

```
def quick_select(L, k):                             <- 1
    p = L[0]                                        <- 2
    A, B, M = [], [], []                            <- 3

    for x in L:                                     <- 4
        if p > x: A.append(x)                       <- 5
        elif p < x: B.append(x)                     <- 5
        else: M.append(x)                           <- 5

    if len(A) > k: return quick_select(A, k)        <- 6
    elif len(A) + len(M) < k:                       <- 7
        return quick_select(B, k - len(A) - len(M)) <- 7
    else: return p                                  <- 8
```

1. 우선, 리스트 L과 정수(자연수) 값인 k를 인자로 받는 함수를 정의한다.
2. L에 있는 값 중에 가장 처음에 있는 값 L[0] 을 p(pivot) 으로 선택한다.
3. p보다 작은 값, 큰 값, 같은 값을 모아서 저장할 자료 구조를 선언한다.
   - 이렇게, 값들을 모아둘 집합들을 모두 빈 리스트로 초기화한다.
4. L에 있는 각각의 원소(값) 을 하나씩 비교하기 위해 반복문을 수행한다.
5. 만약 p와 비교했을 때, x가 작으면 A에, 크면 B에, 같으면 M에 집어넣는다.
6. 만약 A에 들어있는 값의 개수가 k보다 크면, quick_select() 를 다시 호출한다.
   - 이 때, L이 아닌 A에서 k번째로 작은 수를 구하도록 인자를 입력한다.
7. 만약 A와 M에 들어있는 값의 개수가 k보다 작아도, quick_select() 를 호출한다.
   - 이 때, B에서 (k - |A| - |M|) 번째로 작은 수를 구하도록 인자를 입력한다.
8. 만약 앞에 있는 두 가지 조건을 만족하지 않으면, 바로 p를 반환하면 된다.
   - 왜냐하면, M에 들어있는 값들은 모두 p와 같기 때문이다.

<br>

여기서 주의할 점은 재귀적으로 호출된 후에, p가 반환된다는 것이다.

- 따라서, 재귀적으로 호출한 부분의 결과도 모두 반환해야 한다.
- 왜냐하면, 재귀적으로 반환된 결과를 다시 반환해야 하기 때문이다.
- 이렇게, 재귀 호출을 수행한 대기 함수로 결과값을 넘겨줘야 한다.

<br>

작성한 의사 코드의 quick_select() 함수는 실제 파이썬 코드이기 때문에 정상적으로 동작한다.

> 물론, 이것은 가장 좋은 코드가 아니지만, 가장 간단하게 작성할 수 있는 코드 중 하나다.

# 2.

이제, 앞에서 살펴본 Quick Select 알고리즘을 예를 들어서 따라가보자.

```
6 5 1 7 9 3 8 10 2
p

n = 9, k = 8

5 1 3 2 6 7 9 8 10
└─ A ─┘ M └─ B ─┘
└───┬───┘ ─-> k
    5      ↑
        (8 - 5) = 3
```

- n = 9, 총 9개의 숫자가 있으며, k = 8, 찾고자 하는 숫자는 8번째로 작은 숫자다.
- 맨 처음에 있는 원소의 값인 6을 pivot 으로 선택한다.
- pivot 인 6보다 작은 값을 모아 A, 큰 값은 B라 하고, A, M, B 순서로 나열한다.
   - 여기서 pivot 보다 작은 숫자는 5, 1, 3, 2이고, 큰 숫자는 7, 9, 8, 10 이다.
   - 이 경우에는 A와 B에 있는 원래 있는 9개 중에 대략 pivot 을 제외하고 반 정도 된다.
   - A도 4개, B도 4개로 예쁘게 반반씩 나뉜 것이다.
- 그런데, 우리가 원하는 것은 8번째로 작은 숫자를 찾는 것이다.
   - 따라서, 3번째 단계인 A와 M에 들어있는 개수는 총 5개다.
   - 그런데 우리가 원하는 것은 8번째이므로, A와 M에는 우리가 찾고 있는 값이 없다.
   - 그러면, B에서 3번째로 작은 값을 찾아야 한다.
   - 왜냐하면, k = 8, A와 M에 있는 값들의 개수가 5개이기 때문에,  
     B의 입장에서는 8 - 5 = 3 번째로 작은 값이 전체에서 8번째로 작은 값이 되기 때문이다.

<br>

따라서, 다시 quick_select 를 재귀적으로 호출한다.

```
quick_select(B, 3)

7 9 8 10
p

Ø 7 9 8 10
A M └ B ┘
└┬┘ -> k
 1  ↑
 (3 - 1) = 2
```

- 그렇게 해서, 이것을 다시 재귀적으로 반복하면, B의 맨 앞에 있는 7이 pivot 이 된다.
- pivot 보다 작은 값이 없어서, A에는 아무 값도 들어있지 않게 된다.
- B에는 pivot 보다 큰 9, 8, 10이 전부 다 모인다.
- 이 경우는 안 좋은 것이다.
- 왜냐하면, 다시 또 pivot 이랑 비교해서 작은 것과 큰 것으로 나눴는데, 전부 다 큰 값으로 모인 것이다.
- 그러면, 내가 원하는 것은 여기서 3번째로 작은 것이다.
- 그런데, A는 아무것도 없고, M은 7밖에 들어있지 않기 때문에,  
  A와 M에 들어있는 값들의 개수가 1개 밖에 없는 것이다.
- 그러면 내가 원하는 것은 다시 B에서 3 - 1 = 2 번째로 작은 값을 골라야 한다.
- 그러면 B에서 다시 똑같은 일을 반복하는 것이다.
- 맨 처음에 9개에서 1번 내려왔을 때, 9개에서 찾아야 하는 것을 4개에서 찾아야 하는 것으로 바꿨다.
- A와 M에는 우리가 찾는 값이 없으니까, 후보에서 제외할 수 있다.
- 그러니까 B에 있는 4개 중에서만 보면 되는 것이다.
- 그 다음에 한 번 더 재귀적으로 계산해보면, 4개 중에서 찾는 문제가 3개 중에서 찾는 문제로 바뀌었다.
- 앞에서는 9개 중에 반 정도는 버리고, 반에서만 찾아도 되었는데,  
  두 번째에는 4개 중에서 pivot 하나만 제외하고 나머지 다 에서 또 찾는, 좀 더 좋지 않은 상황이 된 것이다.
- 그렇게 해서 뭔가 이것은 내가 고른 pivot 의 값의 크기에 따라서  
  예쁘게 반반씩 나뉠 수도 있고 한쪽에 몰릴 수도 있고, 그런 상황이 번갈아가면서 나타나는 것이다.

<br>

- 그러면, 가장 안 좋은 worst case 는 무엇일까?
- worst case 는 말 그대로 p가 있으면, 한 쪽에 다 몰려서 나눠지는 것이다.
- (n - 1) 개가 전부 다 B에 속하는 것이다.
- 다시 이것을 재귀적으로 했는데, 이번에는 (n - 2) 개가 전부 다 pivot 보다 작은 것이다.
- 그 다음에 또 했는데, 또 (n - 3) 개가 pivot 보다 작다.
- 그 다음에 다시 또 했는데, 이번에는 (n - 4) 개가 pivot 보다 크다.
- 이런식으로, 극단적으로 불균형하게 나뉘어지는 것이다.
- 그러면 이것을 반복할 때마다, 하나씩 밖에 제외가 안된다.
- n개에서 찾는 것이 (n - 1) 개에서 찾는 것이 되고,  
  (n - 1) 개에서 찾는 것이 (n - 2) 개에서 찾는 것이 되고...
- 이런식으로, 계속 반복되는 것이다.

그래서, 이렇게 worst case 에 관련된 것을 점화식으로 써보면, 이렇게 된다.

T(n) 은 n개의 숫자에서 k번째로 작은 값을 찾을 때 걸리는 quick_select 의 비교 횟수라고 하면,  

- worst case(w) 인 경우에는, 이것을 한 번 수행할 때, 여기에 있는 pivot 과 비교를 한다.
- 이렇게 pivot 이 있었으면, (n - 1) 개와 전부 비교한다.
- 이렇게 pivot 보다 작은지 큰지를 봐야 하는데, 전부 다 pivot 보다 작아서 A에 다 속한 것이다.
- 한 번씩 다 비교를 해야 하므로, (n - 1) 번 비교해야 한다.
- 이 때, 단순히 표현하기 위해 n번이라고 할 것이다.
- 그래서 n번을 비교해서 여기있는 숫자를 하나만 줄인 것이다.
- 그리고 (n - 1) 개에 대해서 다시 똑같은 일을 해야 한다.
- 그러면 그 (n - 1) 에서 똑같은 일을 할 때 필요한 비교 횟수는 T(n - 1) 이다.
- 그래서 이러한, 이러한 점화식이 만들어진다. T(n) = T(n - 1) + n
- 이 점화식은 여러분들이 직접 해본다. 쉽다.
- 그러면 이것은 예를 들어서 T(1) = 1 이라고 하면, T(n) = 1 + 2 + .. + n 이 된다.
   - 1부터 n까지 더하는 식으로 전개가 된다.
- 1 + 2 + .. + n = (n * (n + 1)) / 2 이고, 결국, 이것은 O(n^2) 이 된다.
- 즉, 최악의 경우에 n^2 번은 비교를 해야 한다는 것이다.
- pivot 을 너무 거지같이 골라서, 항상 불균등하게 나눠지는 것이다.
- 이렇게 불균등하게 나눠지는 것이 계속해서 반복되는 것이다.
- 그런데, 사실, 실제로 이런 일이 일어날 확률은 극히 작을 것이다.
- 그래서 실질적으로는 이러한 상황을 고민할 필요는 없지만,  
  이론적으로는 worst case 에 n^2 번 비교하는 어마어마한 비교 시간이 걸린다라는 것을 이해해야 한다.

<br>

best case 는 예쁘게 나뉘어 지는 것이다.

```
n = 2 ^ k

T(n) = T(n / 2) + n
     = (T(n / 2 ^ 2) + (n / 2)) + n
       ...
     = n + (n / 2) + (n / 2 ^ 2) + .. + T(n / 2 ^ k)

T(1) = 1

T(n) = (n / 2 ^ k) + (n / 2 ^ (k - 1)) + ..  + (n / 2) + n
     = n * (1 + (1 / 2) + (1 / (2 ^ 2)) + ... + (1 / (2 ^ k)))
     <= 2n
```

- pivot 이 있으면, 맨 처음에 위에서 했던 것처럼 pivot 보다 작은 값들과 큰 값들이 약 (n / 2) 개 정도.
- 예를 들어서, B에서 찾기로 결정했으면, 작은 값들과 pivot 은 확인하지 않아도 된다.
- B에서도 pivot 을 기준으로 작은 값들과 큰 값들이 약 (n / 4) 개이고, 또, 둘 중에 하나에서 찾는 것이다.
- 이번에는, A에서 찾는다고 그러면, 여기서도 pivot 보다 작은 값들과 큰 값들이 약 (n / 8) 개 정도. 
- 이런식으로 매 번, 예쁘게 반반씩 쪼개지는 경우다.
- 그러면, 이런 경우에 T(n) 은 pivot 을 기준으로 나누는데,  
  worst case 에서 처럼 n 번의 비교를 하고, 그 다음에, 재귀적으로 (n / 2) 개 중에서만 값을 찾으면 된다.
- 원래는 n개 중에 찾아야 하는데, 1번 했더니 너무 재수가 좋아서, 그중에 반 정도에서만 찾으면 되는 것이다.
- 이것을 계속 반복한다고 쳐보자.
- 지난 번에 했던 것처럼 비슷하게, n = 2 ^ k 라고 가정을 하고, 이 점화식을 풀어보자.
- 이것을 계속 반복한다고 치면, T(n) = n + (n / 2) + (n / 2 ^ 2) + .. + T(n / 2 ^ k) 이 될 것이다.
- n = 2 ^ k 이기 때문에, (n / 2 ^ k) = 1 이 된다.
- T(1) = 1 이라고 하면, 결국, T(n) = (n / 2 ^ k) + (n / 2 ^ (k - 1)) + ..  + (n / 2) + n 이 된다.
- 그런데 이것은 n을 모두 공통으로 꺼내고, 거꾸로 써서,  
  T(n) = n * (1 + (1 / 2) + (1 / (2 ^ 2)) + ... + (1 / (2 ^ k))) 이 될 것이다.
- 그런데 (1 + (1 / 2) + (1 / (2 ^ 2)) + ... + (1 / (2 ^ k))) 은, 아무리 더해도 2를 넘지 못한다.
- 왜냐하면, 1 에 (1 / 2) 을 더하고, 나머지 반의 반을 더하고, 반의 반을 더하고, ..  
  를 반복하여 아무리 많이 더해도, 2를 넘지 못하기 때문이다.
- 1m 떨어진 벽에 다가가기 위해, (1/ 2)m 만큼 이동하고, 남은 (1/ 2)m 의 반을 이동하고,  
  남은 (1/ 4)m 의 반을 이동하고, .. 이렇게 영원히 가봤자, 1m 떨어진 벽까지 도달할 수 없다.
- 따라서, ((1 / 2) + (1 / (2 ^ 2)) + ... + (1 / (2 ^ k))) 은 1이 안된다.
- 그렇기 때문에 (1 + (1 / 2) + (1 / (2 ^ 2)) + ... + (1 / (2 ^ k)))도 2가 안된다.
- 그러면 결국 T(n) 은 2n 을 넘지 않으므로, O(n) 이 된다.

이렇게 best case 에는 O(n) 이고, worst case 에는 O(n ^ 2) 이다.

- 그러니까, 극과 극이 다르고 너무 차이가 난다.
- 너무 재수가 좋으면, O(n) 만에 굉장히 빠르게 찾을 수 있지만,  
  너무 재수가 안 좋으면, O(n ^ 2) 이나 걸리는 것이다.
- best 와 worst 의 중간인 average case, 재수가 좋고 나쁜 것이 계속 섞여서 나오는 경우,  
  일반적인 경우, 평균적인 경우를 계산해보니, O(n) 이다. 라는 것을 증명할 수 있었다.
- 이 증명은 구름과 강의 노트에 들어있으니, 필요한 사람은 보도록 하면 된다.
- 그런데, 우리 강의의 범위를 벗어나기 때문에,  
  구체적으로 average case 가 O(n) 이 되는 것을 증명하진 않을 것이다.
- 그렇지만, 평균적으로는 O(n) 으로, best case 와 똑같아서, 굉장히 빠른 것이다.
- quick_select 는 worst case 에는 O(n ^ 2) 이지만, best case 와 average case 가 O(n) 으로 상당히 빠르다.
- 그래서, 실제 구현해서 돌려보면, 다른 선택 알고리즘보다는 굉장히 빠르게 돌아가는,  
  실전에 강한 알고리즘이 된다.
