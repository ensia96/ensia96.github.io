# 1. 정렬

컴퓨터 과학에서 가장 잘 알려진 알고리즘 문제 중 하나는 바로 **'정렬(Sorting)'** 인데,  
이름이나 숫자와 같은 요소들을 정렬하는 등 컴퓨터는 항상 정렬 작업을 수행한다.

가장 저렴한 항공료 찾기, 받은 순서대로 메일 나열하기, 성별로 연락처 분류하기 등  
이렇게 일상 생활 속에서 정렬이 필요한 경우를 쉽게 찾아볼 수 있다.

'정렬은 쉬운 것 같은데, 알고리즘이 많아 봐야 얼마나 많겠어?' 라고 생각할 수도 있지만,  
컴퓨터 과학자들이 수 십년 동안이나 발명해온 만큼, 수많은 정렬 알고리즘이 존재한다.

> 버블 정렬(bubble sort), 스파게티 정렬(spaghetti sort) 등 이름들도 참 멋있다.

# 3. 큰 O 표기법

선택 정렬 알고리즘의 의사 코드(Pseudo-Code) 를 살펴보자.

```
function selectionSort(array)
    for i = 0 to end of array

        smallest = i
        for index = i + 1 to end of array
            if array item index < array item smallest
                smallest = index
            end if
        next

        swap array items at inext and smallest
    next
return array
```

- 이렇게 한 번 작성해둔 함수는 다른 항목의 정렬에서도 사용할 수 있다.
- 배열의 맨 위에서 맨 아래까지, 각 위치마다 가장 작은 수를 찾을 때까지 반복된다.
- 이 때, 코드를 살펴보면 for 루프 내부에 다른 for 루프가 포함되어 있다.
- N 개의 항목을 정렬하는 경우의 동작을 대략적으로 정리하면 아래와 같다.
   - 정렬이 필요한 N 개의 항목에 대해 반복한다.
   - 그 안에서 다시 N 개의 항목에 대해 반복한다.
   - 따라서, 총 N * N (혹은 $N^2$) 회 반복된다.

<br>

입력하는 수의 크기와 실해오디는 단계의 수 사이의 관계는 선택 정렬 알고리즘의 복잡도를 특징화한다.

그것은 얼마나 빨리 또는 느리게 알고리즘이 실행될 지 근사치를 제공한다.

컴퓨터 과학자들은 이러한 성장 순서를 '큰 O 표기법(Big-O notation)' 으로 기록한다.(농담 아니다.)

N 제곱은 특별히 효율적이지 않다.

위에서 살펴본 예제 배열에는 n = 8 개 항목이 있으므로 8의 제곱은 64이다.

배열의 크기를 8에서 80으로 늘리면, 실행 시간은 이제 80의 제곱인 6400이 된다.

비록 배열은 단지 8에서 80으로 10배 증가했지만 실행 시간은 100배 증가했다! (64 ~ 6400)

이것의 영향은 배열이 커질수록 확대된다.

Google 과 같은 회사에게는 큰 문제다.

그들은 수백만 개 또는 수십억 개의 항목으로 배열을 정렬해야 한다.

아마, 급성장하는 컴퓨터 과학자로서 더 효율적인 정렬 알고리즘이 있을 지 질문할 것이다.

# 4. 합병 정렬

다시 예전의 분류되지 않은 배열로 돌아가서 다른 알고리즘인 합병 정렬(merge sort) 을 해보자.

합병 정렬은 첫 번째 작업으로 배열의 크기가 1보다 큰 지 확인한다.

1보다 큰 경우 배열을 두 개로 나눈다.

배열의 크기가 8이므로 크기가 4인 2개의 배열로 쪼개진다.

이들은 크기가 1보다 여전히 크기 때문에 다시 크기가 2인 배열로 나눠진다.

마지막에 각각 1개의 항목인 8개의 배열로 분해된다.

이제 합병이라는 이름을 얻게 된 합병 정렬을 할 준비가 되었다.

처음 두 배열을 시작으로 첫 번째 값이자 유일한 값인 (이 경우 307과 239) 를 읽는다.

239는 더 작으므로 먼저 그 값을 가져온다.

남은 번호는 307 뿐이므로 두 번째 값으로 집어넣는다.

두 개의 배열을 성공적으로 병합했다.

이제 남겨진 쌍들에 대해서 이 과정을 반복하며 그들을 정렬된 순서로 놓는다.

그런 다음 합병 과정이 반복된다.

다시, 첫 번째 두 배열을 취해서 그 안에 있는 첫 번째 숫자를 비교한다.

이번에는 239와 214이다.

214가 제일 작기 때문에 번호를 첫 번째로 가져온다.

이제 두 배열에서 처음 두 숫자를 다시 살펴보자.

239와 250이다.

239가 더 작으므로 그 숫자를 다음으로 가져온다.

다음 두 숫자를 살펴보자.

307과 250.

250이 더 작으니 이것을 가져간다.

마지막으로 307만 남았으므로 끝 부분에 추가하면 된다.

모든 경우에서 두 개의 배열로 시작한다.

개별적으로 정렬하고 이들을 합병하여 더 크게 정렬된 배열로 만든다.

크기가 2인 남아있는 배열들에 대해 똑같은 합병 과정을 반복한다.

이제 크기가 4인 2개의 정렬된 배열이 있다.

이전과 마찬가지로 합병하여 각 배열의 첫 번째 두 숫자를 비교하여 더 낮은 수를 가져온다.

모든 숫자가 합병될 때까지 이 작업을 반복한다. 배열은 완전히 다시 정렬된다!

>
나쁜 소식은 몇 번을 정렬하든 인디애나 폴리스에 가려면 여전히 214 달러를 지불해야 한다는 것이다.

# 5. 복잡도 비교

어쨌든 'Big O' 합병 정렬의 계산 복잡도는 N * logN 이다.

N 은 비교하고 합병하는데 필요한 횟수만큼 발생한다.

이 횟수는 정렬 안의 배열 항목 수와 직접 비례한다.

logN 은 병합 단계의 수에서 나온다.

이 예시에서는 8개의 항목인 배열을 4개로, 2개로, 1개로 쪼개었다. 총 3번 쪼갰다.

이런 식으로 반으로 나누는 것은 항목 수 사이에 로그 관계를 가지고 있다.

2를 밑수로 하는 log 8 은 3과 같다.

항렬의 크기를 2배인 16으로 하면 정렬할 항목 수는 2배가 늘어난다.

2를 밑수로 하는 log 16 은 4이므로, 분할 단계 횟수는 1이 증가한다.

배열의 크기를 8에서 1000배인 8000으로 늘려도 분할 단계 수는 꽤 낮다.

2를 밑수로 하는 log 8000 은 대략 13이다.

그 이상이지만 3보다 훨씬 크진 않고, 약 4배 정도가 커졌다.

그럼에도 불구하고 더 많은 숫자를 정렬할 수 있다.

이러한 이유로 병합 정렬이 선택 정렬보다 훨씬 효율적이다.

그리고 이제는 세라믹 고양이 컬렉션을 이름 순서대로 더 빨리 정렬할 수 있다.

# 6. 그래프 탐색

정말 함께 살펴볼 수 있는 수십 개의 정렬 알고리즘이 있지만 그 대신에,  
선생님이 가장 좋아하는 고전 알고리즘 분야의 문제인 그래프 탐색(graph search)을 함께 해볼 것이다.

그래프는 선에 의해 연결되는 노드 네트워크다.

도시와 그들을 연결하는 도로롤 된 지도와 비슷하다고 생각해도 무관하다.

도시들 간의 경로는 각각 다른 시간이 걸린다.

비용이나 무게로 각각의 선을 이름붙일 수 있다.

이 경우에, 여행하는 데 걸리는 주이다.

# 7. 무차별적 접근

이제 'Highgarden' 에 있는 군대가 'Winterfell' 의 성까지 가장 빠른 경로를 찾고 싶다고 가정해보자.

가장 간단한 방법은 모든 단일 경로를 철저히 조사하고 각각의 총 비용을 계산하는 것이다.

이것은 무차별적 접근(brute force aproach) 이라고 한다.

체계적으로 모든 순열을 검사하여 정렬 여부를 확인함으로써 무차별적 접근 방식을 사용할 수도 있다.

이것은 N! 의 복잡성을 가질 것이다.

N! 이란 노드의 수인 N 부터 시작해서 N 보다 1 작은 수, 그보다 1 작은 수, 계속해서 1까지 곱한 수를 말한다.

N 제곱보다 더 나쁜 방법이다.

그러나 우리는 더 영리하게 할 수 있다.

이 그래프 문제의 고전적인 알고리즘 해법은 컴퓨터 과학의 실습 및 이론에서 가장 위대한 지성인 중 한명이었던  
'Edsger Dijkstra' 가 발명해냈다.

그래서 적당히 'Dijkstra 의 알고리즘' 이라고 이름 붙여졌다.

먼저 'Highgarden' 에서 0의 값으로 시작하는 것을 노드 안에 표시한다.

지금은 다른 모든 도시는 물은표가 표시되어 있는데, 아직 그 곳들로의 도착 값을 알지 못한다.

다익스트라 알고리즘은 항상 값이 가장 낮은 노드에서 시작한다.

이 경우에는 하나의 노드에 대해서만 알고 있으므로, 'Highgarden' 에서 시작한다.

해당 노드에 한 단계 떨어져서 연결되어 있는 모든 노드로의 길을 하나씩 따라가며,  
각 노드에 도달하기 위한 값을 기록한다.

알고리즘의 한 단계가 완료된다.

아직도 'Winterfell' 에 도착하지 못했으므로, 다익스트라 알고리즘을 반복 실행한다.

'Highgarden' 을 이미 확인한 상태에서, 다음 최저 값 노드는 "King's Landing" 이다.

이전과 마찬가지로, 모든 방문하지 않은 연결된 도시로 따라간다.

'The Trident' 선의 값은 5이다.

그러나 'Highgarden' 에서 부터 드는 값을 보기 때문에 총 값을 계산한다.

'The Trident' 는 8 + 5 로 13주이다.

이제 'Riverrun' 으로 가는 오프로드 경로를 따라가본다.

25의 높은 값을 가지고 있으며 총합은 33이다.

그러나 'Riverrun' 안에 10밖에 안되는 더 작은 값을 가진 경로를 발견했음을 알 수 있다.

그래서 새로운 길을 무시하고 이전의 더 나은 길을 지키도록 한다.

이제 "King's Landing" 의 모든 길을 살펴보았고, 'Winterfell' 에 도착하지 못했다.

계속 진행하도록 한다.

다음으로 가장 수가 작은 노드는 'Riverrun' 의 10주이다.

먼저 'The Trident' 로의 경로를 확인한다.

10 + 2, 또는 12의 값을 갖는다.

이전에 발견했던 13의 값보다 약간 나아졌으므로, 'The Trident' 에 대한 경로와 비용을 업데이트한다.

'Riverrun' 에서 'Pyke' 로의 선의 값은 3이다.

10 + 3 은 13이며 이전의 값은 14이므로 'Pyke' 의 경로와 값을 업데이트한다.

여기까지가 'Riverrun' 에서 확인한 모든 경로다.

그 다음은, 짐작할 수 있듯 다익스트라 알고리즘이 다시 반복된다.

다음으로 비용이 가장 낮은 노드는 'The Trident' 이며,  
아직 확인하지 않은 'The Trident' 의 경로는 유일하게 'Winterfell' 로 가는 길이다.

그것은 10의 값을 갖고 있고, 'The Trident' 에 도착하는데 드는 비용인 12를 더해야 한다.

총 값은 22다.

'Pyke' 에서 'Winterfell' 까지 마지막 경로를 확인하면 합은 31이다.

가장 작은 값을 가지면서 군대가 가장 빨리 도착할 수 있는 길을 알아냈다.

이제 "King's Landing" 을 피할 수 있다.

# 9. 다익스트라 변형

다익스트라의 원조 알고리즘은 1956년에 구상되었고, 그래프 안의 노드 수의 제곱만큼의 복잡도를 지녔다.

그리고 이미 논의한 것처럼 제곱된 알고리즘은 절대 좋지 않다.

미국 전체의 도로지도와 같은 커다란 범위의 알고리즘 문제를 해결할 수 없기 때문이다.

다행히, 몇 년 뒤 다익스트라의 알고리즘이 개선되었다.

그래프 안의 노드 수를 로그(노드 수)와 곱하고 라인의 수를 더하는 방법으로 말이다.

이것이 더 복잡해 보이지만, 실제로는 꽤 빠르다.

예제 그래프에 적용해보자.

6개의 도시와 9개의 행. 증명해낸다.

알고리즘은 36개의 루프에서 약 14개의 알고리즘으로 떨어진다.

# 10. 알고리즘에 관하여,

정렬과 마찬가지로, 수많은 장단점을 가진 무한한 그래프 탐색 알고리즘이 있다.

Google 지도로 길을 찾는 것과 같은 서비스를 사용할 때마다 다익스트라 알고리즘은 서버에서 실행되어  
최고의 길을 알아내준다.

알고리즘은 어디에나 있으며 현대 세상은 알고리즘 없이는 불가능하다.

알고리즘의 빙산의 일각만을 다루었지만, 컴퓨터 과학자가 되는 핵심 부분은  
기존 알고리즘을 사용해서 필요에 따라 새로운 알고리즘을 작성하는 데에 있다.

그리고, 저는 여러분이 이 작은 경험이 여러분에게 더 찾고자 함에 흥미를 불러 일으켰길 바랍니다.
