# 0. 시작하기에 앞서,

1960년식 'PDP-1' 은 그래픽을 사용한 초기 컴퓨팅의 좋은 예시다.

- 왼쪽에는 캐비닛 크기의 컴퓨터, 가운데에는 전신타자기, 오른쪽에는 둥근 화면이 있다.
- 당시에는 텍스트 기반 작업과 그래픽 작업이 구분되었기 때문에, 장치들도 독립적이었다.

<br>

사실, 이러한 초기 컴퓨터 화면에 텍스트를 선명하게 렌더링하는 것은 어려운 일이었다.

> 반면에, 타자 내용이 적힌 종이는 훨씬 더 뚜렷한 대비와 해상도를 제공했다.

<br>

초기 컴퓨터 화면의 가장 일반적인 용도는 프로그램의 작업을 추적하는 것이었다.

> 여러 값(values) 과 레지스터(register) 를 추적하여 화면에 표시하는 등

- 전신타자기를 이용해 이런 내용을 반복해서 인쇄한다는 것은 말도 안 되는 일이었다.
   - 속도도 느린 데다가, 엄청나게 많은 종이가 낭비될 것이었기 때문이다.
- 반면에, 화면은 동적이고 빠르게 갱신되어서, 일시적인 값을 표기하는 데에 적합했다.

<br>

프로그램 결과를 출력하는 수단으로 컴퓨터 화면은 거의 고려되지 않았다.

> 대신, 계산 결과는 일반적으로 종이와 같은 다른 영구적인 매체에 기록되었다.

<br>

하지만, 화면은 너무나 유용해서 1960년대 초부터 활발하게 사용되기 시작했다.

# 1. 음극선관

수십 년에 걸쳐, 다양한 디스플레이 기술들이 개발되었다.

> 하지만, 초기의 가장 영향력 있는 기술은 **'음극선관(Cathode Ray Tube)'** 이었다.

<br>

음극선관은 방사체를 이용해, 인광체로 도포된 화면으로 전자를 발사하는 방식으로 작동한다.

- 코팅된 부분에 전자가 닿으면, 아주 짧은 시간(몇 분의 1초 정도) 동안 빛난다.

<br>

전자는 전하를 띤 입자들이기 때문에, 전자기장을 이용해 경로를 조작할 수 있었다.

- 전자를 원하는 위치(상하좌우) 로 유도하기 위해 내부에서 금속판이나 코일을 이용한다.

# 2. 벡터와 래스터

음극선관의 조작 방식으로 그래픽을 그리는 방법에는 두 가지가 있다.

첫 번째는 전자빔(beam) 이 형체(shape) 를 추적(trace) 하도록 하는 방법이다.

- 이러한 방식을 **'벡터 주사(Vector Scanning)'** 라고 한다.
- 빛이 잠깐 유지되므로, 경로를 따라 주사를 반복하면, 꽉 찬 이미지를 생성할 수 있다.  
- 단, 이러한 동작을 빛의 지속이 끊기지 않을 정도로 빠르게 반복해야 한다.

<br>

두 번째는 고정된 경로로 반복적으로 이동하면서, 한 줄씩 주사하는 방법이다.

- 왼쪽 위에서부터 오른쪽 아래로 한 줄씩 이동하는 것을 반복한다.
- 그래픽을 만들기 위해, 이동 중에 특정 지점에서만 전자빔을 켠다.
- 이러한 방식을 **'래스터 주사(Raster Scanning)'** 라고 한다.

<br>

이런 식으로 작은 선분으로 이루어진 도형이나 문자 등의 요소들을 화면에 표시할 수 있다.

# 3. 액정표시장치

결국, 디스플레이 기술이 향상되면서 화면에 선명한 점(dot) 들을 렌더링할 수 있게 되었다.

> 이러한 점들은 '화소(pixel)' 라는 이름으로 불리기도 한다.

<br>

오늘날의 **'액정표시장치(Liquid Crystal Display)'** 는 전혀 다른 기술이다.

- 하지만, 이러한 액정표시장치에서도 래스터 주사 방식이 사용된다.
- 아주 작은 빨강, 초록, 파랑 화소의 밝기를 초당 여러 번 갱신하는 방식이다.

<br>

흥미롭게도, 대부분의 초기 컴퓨터는 화소를 사용하지 않았다.

> 물리적으로는 가능했지만, 메모리 사용량이 너무 많았기 때문이었다.

<br>

예를 들어, 200 * 200 픽셀 크기의 이미지에는 총 40,000개의 화소가 포함된다.

- 화소마다 1비트씩만 사용해도 이미지는 40,000비트의 메모리를 소비한다.
- 이 때, 1비트는 흑백(grayscale) 이 아니라, 말 그대로 흑(0) 과 백(1) 이다.
- 40,000비트는 'PDP-1' 이 사용하는 RAM 전체의 절반 이상에 해당하는 규모다.

<br>

따라서, 컴퓨터 과학자와 공학자들은 그래픽을 렌더링하는 영리한 기법을 찾아야만 했다.

# 4. 문자 발생기

초기 컴퓨터는 수만 개의 픽셀 대신, 훨씬 적은 수의 문자 격자(grid) 를 저장했다.

> 가장 일반적으로는 25자의 문자들을 80개, 총 2,000자의 문자로 구성된다.

<br>

아스키(ASCII) 등을 사용하여 각 문자를 8비트로 인코딩한다고 가정해보자.

- 이 때, 화면 전체를 텍스트로 가득 채우기 위해, 16,000비트의 메모리가 소비된다.
- 이는 40,000비트을 사용하던 것에 비교했을 때, 훨씬 더 합리적이라고 할 수 있다.

<br>

위와 같은 방식을 사용하기 위해서는, 컴퓨터에 하드웨어를 추가할 필요가 있었다.

> RAM에서 읽어온 문자를 화면에 그리기 위해 래스터 그래픽으로 변환하는 장치다.

- 이러한 장치를 **'문자 발생기(Character Generator)'** 라고 불렀다.
- 문자 발생기는, 근본적으로 최초의 그래픽 카드(Graphic Card) 였다.

<br>

문자 발생기 내부에는 작은 **'읽기 전용 기억 장치(Read Only Memory, ROM)'** 가 있다.

- 이 때, 읽기 전용 기억 장치에 각 문자를 나타내는 그래픽 요소를 저장한다.
   - 이러한 그래픽 요소들은 '점 행렬(Dot Matrix Pattern)' 이라 부른다.
- 그래픽 카드가 문자 'K' 에 해당하는 8비트 코드를 인식했다고 가정해보자.
   - 문자 발생기는 K의 2차원 형태를 화면의 적절한 위치에 래스터 주사한다.

<br>

이를 위해, 문자 생성기는 컴퓨터 메모리의 일부분에 대한 특별한 접근 권한을 가졌다.

- 바로, **'화면 버퍼(Screen Buffer)'** 라고 불리는, 그래픽을 위해 예약된 메모리 영역이다.
- 컴퓨터 프로그램들은 문자를 화면에 렌더링하기 위해 화면 버퍼에 저장된 값들을 조작했다.
   - 이는 컴퓨터 프로그램이 RAM에 저장된 다른 정보들을 다루는 것과 비슷한 방식이다.

# 5. 아스키 아트

이렇게 문자 생성기를 이용하는 방식은 훨씬 적은 메모리를 사용한다.

> 하지만, 컴퓨터 화면에 그려낼 수 있는 유일한 요소는 문자뿐이었다.

<br>

그럼에도 불구하고, 사람들은 **'아스키 아트(ASCII Art)'** 를 통해 창의력을 발휘했다.

- 기본 문자 집합으로 기초적인 가상(pseudo) 그래픽 인터페이스를 만들려고 했다.
- 밑줄(\_), 더하기(+) 등의 기호를 사용해, 기타 기본 모양(상자, 선 등) 을 만들었다.

<br>

하지만 문자 집합은 너무 작아 몹시 정교한 작업을 수행하기 어려웠다.

그래서 다양한 아스키 확장이 이루어져 새로운 반그래픽 문자가 추가되었다.

화면에 보이는 것과 같은 IBM의 CP437은 DOS에서 사용되었다.

일부 시스템에서는 텍스트 색상과 배경 색상을 몇 가지 추가 비트로 정의할 수 있었다.

이는 방금 본 문자 집합에 완전히 구축된 이 DOS 예시와 같은 멋진 인터페이스를 허용했다.

# 6. 애니메이션

문자 생성기는 메모리를 절약할 수 있는 영리한 방법이었다.

그러나 그들은 임의의 모양을 그릴 수 있는 방법을 제공하지 않았다.

이것은 전기 회로나 건축 계획, 지도 등을 그리고 싶을 때 중요하다.

거의 모든 것들이 텍스트가 아니다!

이를 위해, 컴퓨터 과학자들은 메모리를 잡아먹는 픽셀에 의존없이, CRT에서 가능한 벡터 모드를 사용했다.

아이디어는 매우 간단하다.

화면에 그려지는 모든 내용은 일련의 선으로 정의된다. 텍스트는 없다.

텍스트를 그릴 필요가 있으면 선 밖으로 나오면 된다.

행간은 읽지 않는다. 여기에는 선만 있다.

저는 여기서 선을 그으려고 한다.

이 비디오가 폭 200, 높이 100의 데카르트 평면인 것처럼 가정해보자.

왼쪽 위 구석의 (0, 0) 이 원점이다.

초기 벡터 디스플레이 시스템인 Vextrex 에서 다음의 벡터 명령을 빌려 모양을 그릴 수 있다.

먼저 리셋을 하여, 화면을 깨끗이 하고 전자 총으로 된 도면점을 0으로 움직이고 선의 밝기를 0으로 설정한다.

그런 다음 도면점을 (50, 50) 으로 이동하고, 선 강도를 100% 로 설정한다.

강도가 올라감에 따라, 이제 (100, 50) 으로 이동해서 그 다음 (60, 75) 그리고 다시 (50, 50) 으로 되돌아간다.

이제 마지막으로 할 일은 라인 강도를 다시 0%로 되돌리는 것이다.

이렇게 삼각형을 그렸다.

이 일련의 명령은 160비트 단위를 소모하여, 픽셀 값의 거대한 행렬을 유지하는 것보다 훨씬 효율적이다.

문자 생성기로 문자를 저장하고, 그래픽으로 변환하는 방법과 마찬가지로,  
벡터 명령도 메모리에 저장되고 벡터 그래픽 카드를 사용하여 화면에 렌더링된다.

수백 개의 명령을 순차적으로 화면 버퍼에 함께 묶어 복잡한 그래픽을 만드는 데에 사용할 수 있다.

모두 라인으로 만들어졌다.

이 벡터들은 모두 메모리에 저장되기 때문에, 컴퓨터 프로그램은 값을 자유롭게 업데이트 할 수 있으며,  
시간이 지남에 따라 변경되는 그래픽을 허용한다. - 바로 애니메이션이다!

가장 초기의 비디오 게임 중 하나인 'Spacewar' 는 벡터 그래픽을 사용하여 1962년 PDP-1 에 만들어졌다.

그것은 많은 이후 게임들에게 영감을 주었다.

소행성이나 최초의 상업용 아케이드 비디오 게임인 'Computer Space' 같은 게임에 말이다.

# 7. 스케치패드

1962년은 'Sketchpad' 로 인해 거대한 이정표가 되었다.

'Sketchpad' 는 컴퓨터 이용 설계(Computer-Aided Design, CAD) 를 제공하는 대화형 그래픽 인터페이스다.

그것은 가장 초기의 완벽한 그래픽 응용 프로그램 예제로 널리 간주된다.

그리고 그 발명자인 'Ivan Sutherland' 는 이 돌파구로 이후에 튜링상을 수상했다.

그래픽과 상호작용하기 위해 'Sketchpad' 는 최근 발명된 입력 장치인 라이트펜(lightepen) 을 사용했다.

컴퓨터에 선으로 연결되어 있는 스타일러스 펜이다.

펜은 끝 부분에 광센서를 사용함으로써 컴퓨터 모니터의 새로 고침을 감지했다.

새로 고침의 타이밍을 사용하여, 컴퓨터는 실제로 화면 상의 펜의 위치를 파악할 수 있었다.

이 라이트 펜과 거대한 컴퓨터의 다양한 버튼들로 사용자는 선과 다른 간단한 모양을 그릴 수 있었다.

'Sketchpad' 는 이와 같은 작업을 할 수 있었다.

완벽한 평행선 그리기, 같은 길이의 선 그리기, 완벽한 90도 교차점으로 모서리를 직선화,  
심지어 동적으로 모양을 위아래로 스케일링 할 수도 있었다.

종이로 하기 힘든 이러한 것들을, 이제 컴퓨터가 버튼 하나로 할 수 있다!

사용자는 자신이 만든 복잡한 디자인을 저장한 다음 나중에 디자인에 붙여 넣을 수 있었고,  
심지어 다른 사람들과 공유할 수도 있었다.

전체 도형의 라이브러리를 가질 수 있었다.

전자 구성 요소와 가구 조각처럼 자신의 창조물에 딱 들어맞게 조작할 수 있다.

이것은 오늘날의 관점에선 꽤 일상적으로 들릴지도 모른다.

그러나 1962년, 컴퓨터가 여전히 캐비닛 크기로 천공 카드를 통과하던 때,  
'Sketchpad' 와 라이트 펜은 정말 대단했다.

그들은 컴퓨터 사용법에 대한 중요한 전환점을 나타냈다.

그들은 더 이상 닫힌 문 뒤에서 수학 계산을 하는 기계가 아니었다.

이제 그들은 잠재적 조수로서, 대화식으로 인간의 작업을 보강한다.

# 8. 비트맵 디스플레이

최초의 컴퓨터와 진정한 픽셀 그래픽 디스플레이는 1962년대 후반에 등장했다.

메모리의 비트가 화면 상의 픽셀에 직접 매핑된다.

이는 비트맵 디스플레이라고 불리는 것이다.

전체 픽셀 컨트롤을 사용하면 완전히 임의적인 그래픽이 가능했다.

화면의 그래픽을 거대한 행렬의 픽셀 값으로 여길 수 있다.

이전처럼, 컴퓨터는 프레임 버퍼(Frame Buffer) 라고 하는 픽셀 정보를 위한 특별한 메모리 영역을 따로 둔다.

초기에는 컴퓨터의 RAM이 사용되었지만, 이후의 시스템은 고속 접근을 위해  
그래픽 카드 자체에 위치한 특수 고속 비디오 RAM, 또는 VRAM 을 사용했다.

이것은 오늘날까지 이루어진 방식이다.

8비트 회색도 화면에서 검정색인 0의 강도에서 흰색인 255의 강도까지 값을 설정할 수 있다.

실제로, 그것은 녹색이거나 주황색일 수 있다.

많은 초기 디스플레이는 흰색을 할 수 없었다.

# 9. 그래픽 라이브러리

이 동영상을 매우 저조한 해상도의 비트맵 화면으로 바꾸어보자.

60 * 35 픽셀의 해상도다.

(10, 10) 픽셀을 흰색으로 설정하려면 다음과 같은 코드를 사용하면 된다.

```
framebuffer[10][10] = 255
```

선을 그리고 싶다면, (30, 0) 부터 (30, 35) 까지 루프를 사용할 수 있다.

```
for y = 0 to 35
    framebuffer[30][y] = 255
next
```

그리고 이것은 전체 픽셀 라인을 흰색으로 변경한다.

좀 더 복잡한 것을 그리기 원한다면, 예를 들어 직사각형일 때에는 네 개의 값을 알아야 한다.

```
x = 2
y = 10
width = 4
height = 8
color = 255
```

시작 모서리의 x, y 좌표, 폭과 높이를 알아야 한다.

지금까지 우리는 모든 것을 흰색으로 그렸으니 이 사각형을 회색으로 지정하자.

```
x = 2
y = 10
width = 4
height = 8
color = 127
```

회색은 0에서 255 사이의 중간이므로 127의 색상 값이다.

그런 다음 두 개의 루프 - 하나는 다른 루프에 중첩되어 있으므로  
외부 루프가 반복될 때마다 내부 루프는 한 번 실행되어 사각형을 그릴 수 있다.

```
x = 2
y = 10
width = 4
height = 8
color = 127

for i = x to x + width
    for j = y to y + height
        framebuffer[i][j] = color
    next
next
```

컴퓨터가 드로잉 루틴의 일부로 코드를 실행하면 지정한 모든 픽셀에 색상이 지정된다.

이것을 이렇게 'draw rectangle' 함수로 마무리하자.

```
function drawRectangle(x, y, width, height, color)
    for i = x to x + width
        for j = y to y + height
            framebuffer[i][j] = color
        next
    next
return
```

이제 화면의 다른 두 쪽에 두 번째 사각형을 그리기 위해서, 이번에는 검은색으로 해볼 것이다.

- 이 때, 직사각형 그리기 함수를 호출할 수 있다.

```
drawRectangle(54, 4, 4, 8, 0)
```

우리가 얘기했던 다른 그래픽 구조와 마찬가지로  
프로그램은 프레임 버퍼의 픽셀 정보를 조작하여 대화형 그래픽을 만들 수 있다.

물론 프로그래머는 바닥부터 그리기 함수를 만드는 데에 시간을 낭비하지 않는다.

그들은 즉시 선, 곡선, 도형, 텍스트 등을 그리는 함수를 쓸 수 있는 그래픽 라이브러리를 사용한다.

추상화의 새로운 수준이다.

# 10. 그래픽 인터페이스에 관하여,

비트맵 그래픽의 유연성은 상호작용적 컴퓨팅의 전체적인 새로운 가능성을 열었다.

하지만 수십년간 너무 가격이 비쌌다.

지난 수업에서 이야기했듯이, 1971년 말까지 약 7만 대의 전기 기계식 텔레타이프 기계와  
7만 대의 단말기가 미국에서 사용된 것으로 추정되었다.

놀랍게도, 미국에 약 1,000대의 컴퓨터 만이 대화형 그래픽 화면이었다.

그건 별로 많은 것이 아니다.

그러나 'Sketchpad' 및 'Spacewar' 와 같은 선구적인 노력의 도움을 받아 무대가 설치되었다.

컴퓨터 디스플레이가 유비쿼터스화 되고, 그들과 함께 대화형 그래픽 사용자 인터페이스의 새벽이 열렸다.

이는 다음 몇 가지 강의에서 다루겠다.
