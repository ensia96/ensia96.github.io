# 1. 데이터 분석가가 바라보는 매력적인 이력서의 공통점

> 발표자 : 원티드랩 데이터분석가 이상인

> 주제 : 데이터 관련 업무에서 중요하다고 생각하는 점들을 잘 어필하는 매력적인 이력서와 포트폴리오

## 1-1. 데이터 분석가에 대한 정의

- 데이터 분석가, 사이언티스트, 엔지니어 등 데이터 관련 직군은 그 경계가 모호함
- 실제 현업에서 일하는 사람들도, 본인만의 기준으로 각 직군의 역할을 나누기도 함

<details><summary>데이터 사이언티스트와 데이터 분석가</summary>

- 데이터 사이언티스트는 서비스 목적에 맞게 모델을 생성 또는 고도화하는 직무임
- 데이터 분석가는 사업/제품 분석을 통해, 의사결정에 필요한 자료 제공하는 직무임
- 데이터 분석은 모델 관련 작업에 비해 호흡이 짧고, 피드백이 빠르게 오가는 업무임

| 데이터 사이언티스트 | 데이터 분석가 |
|---|---|
| - 데이터 엔지니어 의존도가 비교적 낮음<br>- 모델링 생성 및 고도화<br>- 호흡이 긴 업무 | - 서비스/제품 분석 전반 참여<br>- 비교적 업무 호흡이 짧음 |

</details>

## 1-2. 데이터 분석가가 하는 일

```
- 타 부서에서의 데이터 추출/분석
- 가설에 기반한 데이터 분석
- 서비스 개선을 위한 제품/사업 분석
- 대시보드 생성 및 관리
- 분석 툴, SQL 활용 가이드(및 교육) 등
```

- 기업의 데이터 이해도, 비지니스 유형, 데이터 보유량에 따라 업무가 달라짐
- 기술만 놓고 보면 그렇게 어렵지 않은 편이기 때문에, 누구나 할 수 있을 것임
- 프로그래밍 언어는 R, Python, 대시보드는 tableau, Power BI 등을 사용함
- 하지만, 이러한 기술적인 부분 외에도 데이터 분석가가 갖춰야 할 요소가 있음

<details><summary>데이터 추출 및 분석</summary>

> 데이터를 추출하여, 타 부서에 전달하는 업무

- 사내에서 쓰이는 분석 툴(GA, amplitude 등) 로 확인할 수 없는 부분이 요청됨
- 최신화나 꾸준한 관찰이 필요한 경우, 쿼리 생성, 자동화 등의 작업을 진행함  
  `(SQL, airflow, docker 등의 기술이 활용됨)`
- 일부 기업의 경우, 이러한 데이터 추출 관련 업무만 수행하는 직군도 있다고 함

<details><summary>데이터 분석가가 신경 써야 하는 부분</summary>

```
- 요청자의 의도를 만족할 수 있는가?
- 이외에 의사결정에 도움이 될만한 추가 데이터, 과거 인사이트는 무엇이 있는가?
- 추가 분석이 필요한가?
- 의사결정자가 제공된 데이터를 바탕으로 논리적인 인사이트를 도출하는가?
```

</details>

</details>

<details><summary>서비스 개선을 위한 제품/사업 분석</summary>

- 제품 또는 사업의 변화 이후, 해당 변화가 서비스에 어떤 영향을 주었는지를 분석함
- 작업 진행 순서는 보통 기획, 개발, 배포로 나뉘며, 단계마다 분석가의 역할이 다름
- 목표를 달성하지 못한 경우, 데이터를 분석해 정확한 원인을 찾기 위해 가설을 세움
- 이렇게 세운 가설을 검증하여 원인을 밝혀내고, 분석 결과와 함께 개선안을 제시함

<details><summary>데이터 분석가가 신경 써야 하는 부분(제품 개발)</summary>

```
기획(개발 전)

- 프로젝트 목적에 맞게 지표를 설정했는가?
- 성과를 보기에 더 적합한 지표, 추가해야 할 지표는 없는가?
- 상위 KPI와 잘 부합하는가?
- 해당 프로젝트의 가치는 얼마나 되는가?

개발

- 성과 측정을 위한 데이터를 기록하는가?

배포(개발 후)

- 배포 후 얼마나 개선되었는가?
- 유의미한 개선이었는가?
- 배포 외에 다른 변수가 없는가?
- 퍼널에 병목이 있는가?
- 유저 그룹별로 지표 변화가 있는가?
```

</details>

</details>

<details><summary>대시보드 생성 및 관리</summary>

- 회사 내의 여러 구성원이 지표의 흐름이나 목표 KPI를 꾸준히 추적하는 데 필요함
- 이러한 대시보드는 분석 팀의 판단 또는 다른 팀에서의 요청을 기반으로 생성됨
- 대시보드를 만들기 위해선 요청/제작자 간의 소통, 즉, 많은 인적 비용이 필요함
- 때문에, 분석가는 기존 환경에서 요청 조건을 만족할 수 있는지 먼저 검토해야 함
- 대시보드를 만들 때는 요청된 지표들이 요청 의도를 만족할 수 있을지 검토해야 함
- 완성 후에도, 요청자의 의도를 만족하기 위해, 꾸준히 피드백을 받고 개선해야 함
- 서비스 확장이나 운영 중에 발생하는 이슈에 맞춰서 지속적으로 갱신/관리해야 함

<details><summary>데이터 분석가가 신경 써야 하는 부분(제품 개발)</summary>

```
대시보드 제작 전

- 요청된 지표를 기존 환경에서 확인할 수 있는가?
- 요청된 지표들이 요청 의도를 만족할 수 있는가?

대시보드 제작

- 완성된 대시보드가 요청 의도를 만족할 수 있는가?

관리

- 서비스 확장에 따른 추가 요청
- 메모리 관리, 원천 데이터 변화로 인한 오류 등의 운영 이슈들
```

</details>

</details>


## 1-3. 데이터 분석가의 자세

데이터 분석가는 의사 결정의 기반이 될만한 인사이트와 재원을 제공하는 직무임

> 앞에서 살펴봤듯, 다른 부서에서 요청받은 업무를 주 업무라고 할 수 있을 정도임

<br>

발표자는 피드백을 주고받는 것을 의사 결정에 참여하는 하나의 과정이라고 생각함

- 때문에, 지속적인 관심과 함께 주도적으로 호기심을 가지는 것이 중요하다고 생각함
- 더 나아가, 이러한 관심과 호기심을 통해 인사이트를 창출하려는 자세를 가져야 함

<br>

이 때, 이러한 '데이터 분석가가 갖춰야 하는 자세' 를 '팬심(fan心)' 에 비유할 수 있음

- 팬심을 가진 사람들의 애정과 호기심은 데이터 분석가에게 필요한 부분과 비슷함
- 유퀴즈에 출연했던 예능 프로그램은 열성 팬이 가진 관심과 열정을 예로 들 수 있음
- 이렇게 대상에 대한 이해도와 정보를 통해, 새로운 인사이트를 얻는 능력이 중요함

## 1-4. 매력적인 이력서

발표자는 이러한 팬심을 담은 이력서/포트폴리오가 서류를 볼 때 매력적이라고 생각함

- 이력서에서 이러한 요소들을 어필하려면, 프로젝트 경험을 잘 담아낼 필요가 있음
- 프로젝트 중에 좋은 경험을 했음에도, 어필할 부분을 떠올리지 못하는 경우가 있음

<details><summary>프로젝트 단계별 어필 요소</summary>

- 분석/기획 단계에서는 수집할 데이터를 정의하고 데이터의 가치에 대해 신경 써야 함
- 프로젝트의 논리적 흐름을 설명하려면, 분석 방법론의 개념보다 선정 이유에 집중해야 함
- 분석 결과와 시각화를 통해 어떤 인사이트를 도출했는지 정리하는 것도 아주 중요함
- 주어진 데이터와 분석 방법론을 통해 단순히 결론을 내린 프로젝트는 좋다고 하기 힘듦
- 프로젝트를 시작하게 된 계기와 프로젝트를 통해 도출한 결론을 정리하는 것이 중요함

```
분석 기획 & 데이터 수집

- 해당 주제를 왜 선정했는가?
- 해당 이슈를 해결했을 때, 누가 어떤 영향을 받게 되는가?
- 어떤 데이터를 활용했고, 어떤 방법으로 수집했는가?

전처리 & EDA

- 전처리 과정을 선정한 근거는 무엇인가?
- EDA 과정에서 어떤 가설을 세웠고, 어떤 분석 결과를 기대했는가?

분석

- 분석 방법론을 선정한 이유는 무엇인가?

분석 결과 & 시각화

- 대시보드를 보는 대상은 누구인가?
- 수집한 자료를 통해 어떠한 인사이트를 도출할 수 있는가?
- 분석 기획에서 예상했던 내용과 실제 내용에는 어떤 차이가 있는가?
```

</details>

## 1-5. 마무리

데이터 사이언티스트라고 해서, 무조건 머신러닝과 딥러닝을 해야 하는 것은 아님

- 채용 공고를 살펴보면, 기업마다 각 직군에 요구하는 역량이 다르다는 것을 알 수 있음
- 기업마다 직군에 대한 정의가 다르므로, 데이터 직군 관련 공고는 모두 확인하길 권장함

# 2. 데이터 과학의 완성, Data Engineering

> 발표자 : SK(주) C&C 데이터 플랫폼 개발자 김현중

> 주제 : 데이터 엔지니어링에 대한 소개, 데이터 엔지니어가 추구하는 방향성

## 2-1. 자기소개

발표자들을 보면 알 수 있듯, 전공에 상관없이, 관심 분야에 따라 직무를 결정할 수 있음
- 물론, 발표자는 고등학교 때부터 코딩을 시작해서, 컴퓨터 과학 역량이 조금 있기는 했음
- 발표자는 통계학 전공이기 때문에, 취업 준비 과정에서는 분석 역량을 강점으로 내세웠음

<br>

사용하는 기술이 굉장히 다양한데, 취업 준비 중에는 주로 파이썬과 분석 도구 위주였음

- 발표자의 경우, 데이터 분석 직군으로 취업했지만, 실제로는 조금 다른 업무를 하고 있음
- 데이터 파이프라인과 모델 서빙, 이 두 가지가 데이터 엔지니어링의 핵심이라 할 수 있음

## 2-2. 취업 준비

발표자가 취업 준비 과정에서 중요시했던 주제는 '프로젝트 경험' 과 'T자형 인재' 였음

- 프로젝트 내용과 완성도보다, 프로젝트 진행 과정에서 있었던 일과 배운 점이 더 중요함
- 이러한 이유로, 발표자는 프로젝트나 포트폴리오의 주제는 크게 중요하지 않다고 생각함
- 발표자가 제시했던 키워드 중 하나, 'T자형 인재' 는 '넓은 배경지식을 가진 전문가' 를 뜻함
- 이를 어필하기 위해, 발표자는 동아리나 대외활동 등 다양한 경험이 있는 인재임을 강조함

## 2-3. 담당 업무

> 불필요한 홍보 내용은 생략

<br>

발표자는 모델링, 시각화, 전처리 등을 자동화하는 데이터 분석 플랫폼을 개발하고 있음

- 'Pipeline', 'Modeler', 'Multi-Cluster Manager' 등의 서비스로 구성된 솔루션 플랫폼임
- 그중에서도 분석 환경 도출, 코드 관리, 모델 배포하는 'Modeler' 서비스를 담당하고 있음

<br>

담당자의 현재 직무(표면적인 직무) 는 AI Engineer 인데, 이는 굉장히 모호한 직무임

- 이는 인공지능 관련 요소를 개발하는 직무이지만, 실제로는 다른 업무를 하고 있음
- 실제 업무는 플랫폼 개발과 SI 개발이며, 팀 내에서 서비스 개발자 역할을 맡고 있음 

<br>

발표자의 회사는 SI 업읊 주로 해서, 개발 대상도 플랫폼보다는 판매용 서비스에 가까움

- 데이터 분석가로 입사했지만, 서비스 개발과 데이터 분석을 연결하는 업무를 맡고 있음
- 발표자는 풀스택 개발자로서 작업하고 있어서, 업무 범위가 상당히 넓은 편이라고 함

<br>

최근에 진행한 업무로는, '분석 도우미 개발' 과 '모델 서빙 API 제공' 등이 있다고 함

- 여기서 분석 도우미는 데이터 시각화와 모델 비교 기능을 제공하는 파이썬 패키지임
- 모델 배포, 배포된 모델 관리, 요구 수집/처리 등의 작업을 주로 담당하고 있다고 함
- 데이터 수집, 저장, 처리, 인프라 구성이 데이터 엔지니어의 주 역할이라 할 수 있음

## 2-4. 데이터 처리

데이터 처리는 'Extract', 'Transform', 'Load' 로 구성되며, 이를 'ETL' 이라고 부름

- 이러한 세 가지 요소는, 실제로 데이터베이스에서 데이터를 가져올 때 거치는 과정임
- 추출(Extract), 변형(Transform), 적재(Load) 하는 부분까지가 데이터 처리의 영역임

<br>

데이터(빅데이터) 처리 과정은 데이터 수집, 저장, 처리, 분석, 시각화 단계로 구성됨

- 데이터 엔지니어링은 데이터 수집, 저장, 처리와 인프라 구성을 포함하는 분야임
- 풍문으로는, 데이터 분석을 데이터 엔지니어링의 일부로 보는 경우도 있다고 함

<br>

각 단계를 컴포넌트로 볼 수 있으며, 이러한 컴포넌트가 연속적으로 이어져 있음

- 데이터가 파이프의 물처럼 흘러가기 때문에, 이를 데이터 파이프라인이라고 부름
- 이렇게 나눠진 각 단계의 내부에서도, 정보 추가/변형 등 여러 단계로 나뉠 수 있음
- 컴포넌트마다 정해진 순서가 있으며, 그 순서를 따라야만 데이터가 제대로 정제됨

<br>

하나의 컴포넌트에서 문제가 생겨도, 다른 컴포넌트에 영향을 주지 않도록 해야 함

- 각 컴포넌트를 독립적으로 구성하면, 이렇게 컴포넌트 간의 문제 전파를 막을 수 있음
- 데이터 엔지니어는, 이러한 대응을 통해 파이프라인이 문제없이 동작하도록 해야 함
