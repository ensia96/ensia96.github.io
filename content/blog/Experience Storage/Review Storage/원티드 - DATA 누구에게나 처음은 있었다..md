# 1. 데이터 분석가가 바라보는 매력적인 이력서의 공통점

> 발표자 : 원티드랩 데이터분석가 이상인

> 주제 : 데이터 관련 업무에서 중요하다고 생각하는 점들을 잘 어필하는 매력적인 이력서와 포트폴리오

## 1-1. 데이터 분석가에 대한 정의

- 데이터 분석가, 사이언티스트, 엔지니어 등 데이터 관련 직군은 그 경계가 모호함
- 실제 현업에서 일하는 사람들도, 본인만의 기준으로 각 직군의 역할을 나누기도 함

<details><summary>데이터 사이언티스트와 데이터 분석가</summary>

- 데이터 사이언티스트는 서비스 목적에 맞게 모델을 생성 또는 고도화하는 직무임
- 데이터 분석가는 사업/제품 분석을 통해, 의사결정에 필요한 자료 제공하는 직무임
- 데이터 분석은 모델 관련 작업에 비해 호흡이 짧고, 피드백이 빠르게 오가는 업무임

| 데이터 사이언티스트 | 데이터 분석가 |
|---|---|
| - 데이터 엔지니어 의존도가 비교적 낮음<br>- 모델링 생성 및 고도화<br>- 호흡이 긴 업무 | - 서비스/제품 분석 전반 참여<br>- 비교적 업무 호흡이 짧음 |

</details>

## 1-2. 데이터 분석가가 하는 일

```
- 타 부서에서의 데이터 추출/분석
- 가설에 기반한 데이터 분석
- 서비스 개선을 위한 제품/사업 분석
- 대시보드 생성 및 관리
- 분석 툴, SQL 활용 가이드(및 교육) 등
```

- 기업의 데이터 이해도, 비지니스 유형, 데이터 보유량에 따라 업무가 달라짐
- 기술만 놓고 보면 그렇게 어렵지 않은 편이기 때문에, 누구나 할 수 있을 것임
- 프로그래밍 언어는 R, Python, 대시보드는 tableau, Power BI 등을 사용함
- 하지만, 이러한 기술적인 부분 외에도 데이터 분석가가 갖춰야 할 요소가 있음

<details><summary>데이터 추출 및 분석</summary>

> 데이터를 추출하여, 타 부서에 전달하는 업무

- 사내에서 쓰이는 분석 툴(GA, amplitude 등) 로 확인할 수 없는 부분이 요청됨
- 최신화나 꾸준한 관찰이 필요한 경우, 쿼리 생성, 자동화 등의 작업을 진행함  
  `(SQL, airflow, docker 등의 기술이 활용됨)`
- 일부 기업의 경우, 이러한 데이터 추출 관련 업무만 수행하는 직군도 있다고 함

<details><summary>데이터 분석가가 신경 써야 하는 부분</summary>

```
- 요청자의 의도를 만족할 수 있는가?
- 이외에 의사결정에 도움이 될만한 추가 데이터, 과거 인사이트는 무엇이 있는가?
- 추가 분석이 필요한가?
- 의사결정자가 제공된 데이터를 바탕으로 논리적인 인사이트를 도출하는가?
```

</details>

</details>

<details><summary>서비스 개선을 위한 제품/사업 분석</summary>

- 제품 또는 사업의 변화 이후, 해당 변화가 서비스에 어떤 영향을 주었는지를 분석함
- 작업 진행 순서는 보통 기획, 개발, 배포로 나뉘며, 단계마다 분석가의 역할이 다름
- 목표를 달성하지 못한 경우, 데이터를 분석해 정확한 원인을 찾기 위해 가설을 세움
- 이렇게 세운 가설을 검증하여 원인을 밝혀내고, 분석 결과와 함께 개선안을 제시함

<details><summary>데이터 분석가가 신경 써야 하는 부분(제품 개발)</summary>

```
기획(개발 전)

- 프로젝트 목적에 맞게 지표를 설정했는가?
- 성과를 보기에 더 적합한 지표, 추가해야 할 지표는 없는가?
- 상위 KPI와 잘 부합하는가?
- 해당 프로젝트의 가치는 얼마나 되는가?

개발

- 성과 측정을 위한 데이터를 기록하는가?

배포(개발 후)

- 배포 후 얼마나 개선되었는가?
- 유의미한 개선이었는가?
- 배포 외에 다른 변수가 없는가?
- 퍼널에 병목이 있는가?
- 유저 그룹별로 지표 변화가 있는가?
```

</details>

</details>

<details><summary>대시보드 생성 및 관리</summary>

- 회사 내의 여러 구성원이 지표의 흐름이나 목표 KPI를 꾸준히 추적하는 데 필요함
- 이러한 대시보드는 분석 팀의 판단 또는 다른 팀에서의 요청을 기반으로 생성됨
- 대시보드를 만들기 위해선 요청/제작자 간의 소통, 즉, 많은 인적 비용이 필요함
- 때문에, 분석가는 기존 환경에서 요청 조건을 만족할 수 있는지 먼저 검토해야 함
- 대시보드를 만들 때는 요청된 지표들이 요청 의도를 만족할 수 있을지 검토해야 함
- 완성 후에도, 요청자의 의도를 만족하기 위해, 꾸준히 피드백을 받고 개선해야 함
- 서비스 확장이나 운영 중에 발생하는 이슈에 맞춰서 지속적으로 갱신/관리해야 함

<details><summary>데이터 분석가가 신경 써야 하는 부분(제품 개발)</summary>

```
대시보드 제작 전

- 요청된 지표를 기존 환경에서 확인할 수 있는가?
- 요청된 지표들이 요청 의도를 만족할 수 있는가?

대시보드 제작

- 완성된 대시보드가 요청 의도를 만족할 수 있는가?

관리

- 서비스 확장에 따른 추가 요청
- 메모리 관리, 원천 데이터 변화로 인한 오류 등의 운영 이슈들
```

</details>

</details>


## 1-3. 데이터 분석가의 자세

데이터 분석가는 의사 결정의 기반이 될만한 인사이트와 재원을 제공하는 직무임

> 앞에서 살펴봤듯, 다른 부서에서 요청받은 업무를 주 업무라고 할 수 있을 정도임

<br>

발표자는 피드백을 주고받는 것을 의사 결정에 참여하는 하나의 과정이라고 생각함

- 때문에, 지속적인 관심과 함께 주도적으로 호기심을 가지는 것이 중요하다고 생각함
- 더 나아가, 이러한 관심과 호기심을 통해 인사이트를 창출하려는 자세를 가져야 함

<br>

이 때, 이러한 '데이터 분석가가 갖춰야 하는 자세' 를 '팬심(fan心)' 에 비유할 수 있음

- 팬심을 가진 사람들의 애정과 호기심은 데이터 분석가에게 필요한 부분과 비슷함
- 유퀴즈에 출연했던 예능 프로그램은 열성 팬이 가진 관심과 열정을 예로 들 수 있음
- 이렇게 대상에 대한 이해도와 정보를 통해, 새로운 인사이트를 얻는 능력이 중요함

## 1-4. 매력적인 이력서

발표자는 이러한 팬심을 담은 이력서/포트폴리오가 서류를 볼 때 매력적이라고 생각함

- 이력서에서 이러한 요소들을 어필하려면, 프로젝트 경험을 잘 담아낼 필요가 있음
- 프로젝트 중에 좋은 경험을 했음에도, 어필할 부분을 떠올리지 못하는 경우가 있음

<details><summary>프로젝트 단계별 어필 요소</summary>

- 분석/기획 단계에서는 수집할 데이터를 정의하고 데이터의 가치에 대해 신경 써야 함
- 프로젝트의 논리적 흐름을 설명하려면, 분석 방법론의 개념보다 선정 이유에 집중해야 함
- 분석 결과와 시각화를 통해 어떤 인사이트를 도출했는지 정리하는 것도 아주 중요함
- 주어진 데이터와 분석 방법론을 통해 단순히 결론을 내린 프로젝트는 좋다고 하기 힘듦
- 프로젝트를 시작하게 된 계기와 프로젝트를 통해 도출한 결론을 정리하는 것이 중요함

```
분석 기획 & 데이터 수집

- 해당 주제를 왜 선정했는가?
- 해당 이슈를 해결했을 때, 누가 어떤 영향을 받게 되는가?
- 어떤 데이터를 활용했고, 어떤 방법으로 수집했는가?

전처리 & EDA

- 전처리 과정을 선정한 근거는 무엇인가?
- EDA 과정에서 어떤 가설을 세웠고, 어떤 분석 결과를 기대했는가?

분석

- 분석 방법론을 선정한 이유는 무엇인가?

분석 결과 & 시각화

- 대시보드를 보는 대상은 누구인가?
- 수집한 자료를 통해 어떠한 인사이트를 도출할 수 있는가?
- 분석 기획에서 예상했던 내용과 실제 내용에는 어떤 차이가 있는가?
```

</details>

## 1-5. 마무리

데이터 사이언티스트라고 해서, 무조건 머신러닝과 딥러닝을 해야 하는 것은 아님

- 채용 공고를 살펴보면, 기업마다 각 직군에 요구하는 역량이 다르다는 것을 알 수 있음
- 기업마다 직군에 대한 정의가 다르므로, 데이터 직군 관련 공고는 모두 확인하길 권장함

# 2. 데이터 과학의 완성, Data Engineering

> 발표자 : SK(주) C&C 데이터 플랫폼 개발자 김현중

> 주제 : 

데이터에 관한 분야이긴 한데, 앞에 발표하신 상인님이 다뤘던 주제와는 조금 다른 부분을 보려고 함

---

개인적인 이야기

발표자는 분석에 가까운 통계학을 전공하고, 데이터 엔지니어로 일하고 있음

전공에 크게 상관없이, 관심 분야에 따라서 직무를 결정할 수 있다는 것을 어느정도 느낄 수 있을 것임

발표자는 코딩을 고등학교 때부터 시작해서, 컴퓨터 과학 역량이 아예 없는 것은 아니었음

일단, 대학 시잘에 주로 공부했던 것은 통계학이었기 때문에, 분석 역량을 강점으로 하여 취업 준비를 했음

2018년도 하반기부터 준비했고, 18개 정도의 회사에 지원해서 최종 합격한 회사인 현재 회사에 재직중임

기술 셋을 보면, 굉장히 다양하다는 것을 알 수 있음

회사에서 새로 알게된 내용도 굉장히 많음

취업 준비할 때는 거의 파이썬이나 분석쪽으로 준비를 했다면, 회사에서 다루는 기술셋도 굉장히 많음

자바스크립트라던지, 처음 들어보실만한 기술이 굉장히 많을 텐데, 도커나 쿠버네티스, AWS, 그리고 기초 서비스 개발을 위한 언어들도 다루고 있음

발표자는 데이터 분석 직군으로 입사를 했는데, 실제로 하는 업무는 조금 다를 수도 있음

특히, 기업 규모가 클수록 이러한 성향이 강함

회사 업무를 하다 보니, 데이터 분석가는 또 다른 엔지니어의 매력을 느끼게 되어서, 현재 관심사는 데이터 파이프라인과 모델 서빙으로 정의해놨는데, 이 두 가지가 데이터 엔지니어링의 핵심이라고 볼 수 있음

뒤에서도 이 부분에 대한 내용은 조금 더 설명할 예정임

---

취업 준비 토픽

발표자는 취업 준비를 할 때 프로젝트 경험과 T자형 인재, 이렇게 크게 두 가지를 준비함

모두가 알고 있듯이, 프로젝틀 경험이 굉장히 중요하게 작용함

어떤 프로젝트를 했고, 얼마나 완성도가 높냐보다는, 얼마나 자기 자신에게 도움이 되었고, 결과보다는 프로젝트를 통해서 얻은 게 무엇인지, 어떤 어려운 점이 있었는지, 그것을 어떻게 해결했는지와 같은 과정이 취업 준비할 때 중요하게 다뤄지고 있어서, 실제로 어떤 프로젝트를 준비해야 되는지, 어떤 포트폴리오를 준비해야 되는지는 크게 중요하지 않다고 생각함

실제로 경험을 해본 바에 의하면 그럼

발표자가 두 번째로 어필했던 것은 T자형 인재임

T자형 인재는 넓은 배경지식을 가진 전문가라고 볼 수 있음

발표자는 다양한 대외활동 경험과 동아리에서 리더를 맡으면서 여러 가지 다양한 경험을 한 인재임을 강조함

실제 업무는 플랫폼 개발과 SI 개발, 두 가지로 나뉨

팀에 주어진 주 역할은 주 역할은 서비스 개발자 역할임

서비스 개발 작업과, 개발한 서비스를 유지보수 해주는 SI 개발로 나뉘어짐

원래 발표자가 다니는 회사는 SI 업을 주로 하던 회사임

그래서 SI 형태의 서비스, 원래 플랫폼은 클라우드 식으로 해서, 온라인으로 제공하는 형식이 주가 되어야 하는데, SI가 가능하다는 형식으로 플랫폼을 판매하고 있음

SI의 장점은 커스터마이징이 가능하다는 점이고, 이러한 장점을 어필하다보니, 실제로 개발해야 하는 양이 상당히 많은 편임

발표자는 분석가로 입사했지만, 사실 팀 내에 분석가는 많지는 않음

실제로, 지금 분석 서비스를 만들고 있는데, 분석 서비스를 만들기 위해서는 분석 지식이 어느정도 필요함

분석가가 어떠한 점을 불편해하는지, 분석 업무와 인프라를 어떻게 엮을 것인지를, 개발을 하기 위해서는 분석 지식도 굉장히 많이 필요한데, 담당자는 서비스 개발과 데이터 분석을 연결하는 업무를 맡고 있음

서비스 개발의 프론트엔드, 백엔드, 데이터베이스 세 가지를 다 할 수 있는 개발자를 풀스택 개발자라고 부르는데, 발표자는 잘하지는 못하지만 세 가지를 다 다루고 있음

서비스의 하나의 기능을 처음부터 끝까지 만들고 있고, 이를 데이터 분석과 어떻게 엮을 것인가하는 업무까지 업무 범위가 상당히 넓은 편임

최근에 했던 업무는 분석 helper 개발과 모델 서빙 API 제공, 이렇게 두 가지가 있음

분석 helper의 경우, 분석 코드의 특정한 코드를 뒤에 추가하면, 그 코드가 여러 가지 서비스 내부에서 시각화하고, 모델을 비교하는 기능을 제공해주는 파이썬 언어에 기반한 패키지를 개발함

플라스크라는 웹 언어를 이용해서 개발된 모델을 외부에 배포하는 그런 작업을 주로 맡아왔음

그 외부에 배포되는 모델을 관리하고, 요구를 수집하고 요구를 처리하는 역할을 맡고있음

분석 외에 플랫폼을 만들어주는 일인 만큼, 특히 엔지니어링적인 요소가 많이 반영되어있음

데이터를 처리하고, 저장하고, 핸들링하고, 인프라 구성까지가 데이터 엔지니어링의 주 역할이라고 볼 수 있음

데이터 엔지니어링이 정확하게 무엇을 다루는지에 대한 답을 설명하기 위해, 원론으로 돌아가서 빅데이터가 무엇인지 간단하게 알아보려고 함

데이터 처리

전통적인 데이터 처리 요소는 'extract', 'transform', 'load', 이렇게 세 가지로 구성되는데, 이를 ETL 이라고 부름

실제로 데이터베이스에서 데이터를 가져올 때 거치는 과정임

extract(추출) 하고, transform(변형) 하고, load(적재) 하는 역할까지가 데이터 처리의 요소임

빅데이터의 특징을 7개의 V로 보여주는 자료를 보면, 그중에서도 가장 중요한 특징이 volume(크기) 와 valocity(데이터 처리 속도) 임

'Real-Time Processing(실시간 처리)', 'Distributed Processing(분산 처리)' 은 앞에서 봤던 ETL 에 비해서, 빅데이터, 진짜 큰 데이터를 분석하고 처리하기 위해서 중요하게 다뤄지는 두 가지 주제임

이 두 가지를 데이터 엔지니어가 주로 핸들링하게 됨

앞에서 봤던 데이터 처리 과정을 구조화하면, 수집, 저장, 처리, 분석, 시각화 단계로 구성됨

데이터 처리와 빅데이터 처리 모두 비슷한 과정을 거치며, 데이터 엔지니어링은 이중에서도 데이터를 수집, 저장, 처리하고, 이러한 과정을 위한 인프라에 걸쳐있는 굉장히 넓은 분야임

발표자가 얼핏 듣기로는, 데이터 분석을 데이터 엔지니어링의 일부로 보기도 한다고 함

데이터 분석과 시각화에 비해서, 생각보다 할 게 많고, 내용이 방대해서 공부하기도 상당히 까다로운 편임

이러한 5단계의 각 요소들을 하나의 컴포넌트라고 볼 수 있음

이러한 컴포넌트를 연속적으로 이어서, 데이터가 파이프의 물처럼 흘러갈수 있도록 만드는 것을 데이터 파이프라인이라고 부름

사실, 앞에서는 5가지 단계로만 설명했지만, 수집, 저장, 처리 단계의 내부에서도 여러 가지 단계가 있을 수 있음

데이터를 처리할 때, 열을 추가하거나 변형하는 등의 작업이 각각 하나의 컴포넌트가 됨

이러한 컴포넌트를 파이프처럼 이어놓은 것이 데이터 파이프라인임

컴포넌트들은 순서가 있으며, 그 순서대로 이어져야 데이터가 정확하게 정제됨

각 컴포넌트는 독립적으로 구성되서, 한 컴포넌트가 죽는다고 하더라도 다른 컴포넌트들이 동작하는 데에는 문제가 없게 하는 것이 데이터 엔지니어들의 숙명임

데이터 과학

데이터 과학 측면에서 데이터 엔지니어링의 가치를 살펴볼 것임

데이터 과학이라고 표현했는데, 데이터 사이언스라고도 부르며, 두 용어는 동의어임

이후의 내용은 발표자의 개인적인 의견이며, 다른 데이터 사이언티스트들의 의견은 다를 수 있음

데이터 과학은 앞에서 살펴봤듯, 데이터 엔지니어링과 그 외의 요소들로 나뉨

데이터 엔지니어링 이외의 요소는 데이터 분석과 시각화로 구분할 수 있음

데이터 과학은 'Data Engineering', 'Data Analysis', 'Data Visualization', 총 3가지 분야로 구분됨

이러한 분야의 역량은 다음과 같음

```
Data Engineering

- Data Pipeline
- Data Service

Data Analysis

- Statistical Analysis
- M.L / D.L

Data Visualization

- Business Intelligence
```

데이터 파이프라인과 데이터 서비스와 같은 플랫폼을 만들어주는 데이터 엔지니어

통계, 분석과 머신러닝, 딥러닝과 같은 모델링을 다루는 데이터 분석가

데이터를 효율적으로 보여주고, 그렇게 잘 시각화된 데이터에서 인사이트를 뽑아내는 데이터 분석가, 데이터 마케터라고도 부름

이와 연관된 학문 분야는 다음과 같음

데이터 엔지니어는 Computer Science, 데이터 분석가는 Math + Statistics, 데이터 마케터는 Business + Design 역량이 굉장히 중요하게 다뤄지는 편임

실제로, 현업 분들도 이러한 전공 분포를 많이 따라감
