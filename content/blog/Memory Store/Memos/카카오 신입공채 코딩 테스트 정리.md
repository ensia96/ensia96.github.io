## 1. 2018 KAKAO BLIND RECRUITMENT(1차)

> 출처 : ['카카오 신입 공채 1차 코딩 테스트 문제 해설'](https://tech.kakao.com/2017/09/27/kakao-blind-recruitment-round-1/)

- 업무에서 있을 만한 상황을 가정, 다양한 분야에서 출제한 비교적 쉬운 수준의 독창적인 문제
- 자료 구조, 알고리즘 등의 전산학 기초를 충분히 학습한 사람은 누구나 풀 수 있을 만한 문제

### 1-1. 비밀 지도

> 비트 연산(Bitwise Operation)

> 정답률 : 81.78%

- 예시에 2진수 처리가 필요함을 암시하는 힌트가 있고, 둘 중 하나가 1인 경우를 처리하면 됨
- if else 구문을 이용해 푸는 방법도 있지만, 비트 연산 활용 능력을 확인하는 것이 문제의 의도

### 1-2. 다트 게임

> 문자열 처리(String Manipulation), 토큰화(Tokenizing) 와 의미 분석(Semantic Analysis)

> 정답률 : 73.47%

- 앞에서부터 한 글자씩 잘라서 처리하거나, 컴파일러처럼 토큰화와 의미 분석을 통해 계산 가능
- 정규식으로 비교적 쉽게 풀 수 있으며, 두 자리 숫자와 음수 점수가 있다는 점에 유의해야 함

### 1-3. 캐시

> LRU 캐시 교체 알고리즘 구현(검색 가능. 단, 출처 명시는 필수)

> 정답률 : 45.26%

- 입출력에 캐시 사이즈 0이 포함되어 있어서, 단순한 코드 복사/붙여넣기만으로는 해결 불가
- 입출력 예제 결과를 분석하면 쉽게 풀 수 있으며, 비정상적인 상황에 대한 예외 처리가 필요

### 1-4. 셔틀버스

> 문자열로 주어지는 시간을 조건에 맞게 계산

> 정답률 : 26.79%

- 일부 조건이 까다로운 편이며, 전체적인 계산은 어렵지 않지만, 정확히 계산하는 것이 어려움

### 1-5. 뉴스 클러스터링

> 자카드 유사도를 계산하는 프로그램 작성 (다중집합(multiset) or 병합 정렬(Merge Sort) 활용)

> 정답률 : 41.84%

- 기본적으로, 문제 풀이에 필요한 요소인 자카드 유사도와 다중집합에 대한 설명이 주어짐
- 일부 언어에서 기본적으로 제공하지 않고, 자주 접하는 자료 구조가 아닌 다중집합을 활용
- 다중집합 자료 구조를 사용하지 않아도, 정렬된 배열과 병합 정렬을 응용하는 풀이법도 있음
- 교집합과 합집합을 잘 구하고, 공집합 나눗셈(division by zero) 에 유의하면 쉽게 풀 수 있음

### 1-6. 프렌즈4블록

> 게임 요구 사항을 구현

> 정답률 : 48.01%

- 블록 매트릭스를 생성하여 스캔하고, 조건에 맞는 블록 덩어리를 제거하는 작업을 반복해야 함
- 제거할 블록이 남아있지 않을 때까지 작업을 반복한 후, 사라진 블록 자리의 수를 계산하면 됨

### 1-7. 추석 트래픽

> 초당 최대 처리량이 되는 구간 윈도우 찾기

> 정답률 : 17.99%

- 전부 스캔하기에는 범위가 너무 넓고, 단위 또한 작아서, 필요한 연산의 수가 너무 많아짐
- 타임아웃이 발생하는 것을 막기 위해, 스캔 범위를 줄일 방법을 도출해내는 것이 중요함
- 이러한 방법은, 각 로그의 시작과 끝에서 요청량이 변한다는 규칙을 통해 알아낼 수 있음
- 이 방법 외에도, 윈도우를 이용하지 않고, O(n log n) 의 시간 복잡도로 푸는 방법도 있음

## 2. 2018 KAKAO BLIND RECRUITMENT(2차)

> 출처 : ['카카오 신입 공채 2차 코딩 테스트 문제 해설'](https://tech.kakao.com/2017/10/24/kakao-blind-recruitment-round-2-2/)

- 1차에서는 자료 구조, 알고리즘 등 전산학 기초 지식과 기본적인 구현 역량을 확인
- 2차는 빠른 습득력, 요구사항 분석과 기능 설계 능력, 모니터링/개선 능력을 확인

> 문제 내용 : 이미지 특징값을 수집하는 Crawler 구현

### REST API와 JSON Parsing

- 우선, 토큰을 발급받은 후에, 테스트 전용 서버로 연결되는 5개 카테고리의 URL 을 받아야 함
- 카테고리의 문서를 읽고, 그 안에 있는 이미지들의 특징값을 추출하여 저장/삭제하는 문제임

### 반복적인 모니터링과 최적화(제한된 토큰 유효시간)

- 토큰 유효기간은 10분으로 설정, 점수 현황판을 공개하여 지원자의 최적화 작업을 유도함
- 문제가 발생할 때마다 로그를 남기고, 모니터링을 통해 원인이나 병목 현상을 찾도록 유도함

### 평가 메트릭 분석 후에 전략 수립

- 누락되거나 삭제되지 않은 이미지, 잘못된 정보의 수에는 패널티가 붙도록 기준을 정함
- 실제 상황에서 얼마나 큰 문제가 될 수 있는지에 따라서 패널티의 크기를 다르게 적용함
- 참가자들이 불필요한 요청의 수를 줄이도록, 전체 쿼리의 양에도 약간의 패널티를 부여함

### 고득점을 위한 병렬 처리

- 가장 자주 호출하게 되는 이미지 엔드포인트의 처리 속도를 조절하여 병렬 처리를 유도함
- 처리 속도는 16ms ~ 120ms 이므로, 네트워크 지연을 포함해 초당 요청 제한은 50건 정도
- 이 때, 테스트 전용 서버의 처리 속도는 배치 처리 개수에 따라서 exp 그래프 형태를 띰
- 따라서, 최대 배치 처리 사이즈인 50개보다, 30 ~ 40개 정보의 요청에서 가장 효율이 좋음

### 예외 처리

- 크롤링 작업 중에 사용자가 이미지를 삭제하거나 서버가 다운되는 경우도 반영되어 있음
- 따라서, 확률적으로 성공하는지, 무조건 실패하는지에 따라 재시도나 블록 처리가 필요함

### 다양한 시나리오에 대응

- 중복 추가, 추가 후 바로 삭제, 없는 이미지 삭제, 추가 삭제 추가가 반복되는 상황도 있음
- 이 때, 모든 이미지를 색인하면서 발생하는 공간 낭비는 간단한 메모리 캐시로도 해결 가능
- 연산 목록은 단순히 set 자료형을 활용해 중복을 상쇄하면 문제가 발생하도록 구성되어 있음
- 또한, 추가/삭제 연산이 발생하는 범위가 넓어서, 시간과 정확성 사이에서 의사결정이 필요

### 카테고리별 특성 파악

- 카테고리마다 추가/삭제 비율과 URL 갱신 시간에 차이가 있어서, 적절한 전략이 필요함
- 카테고리별 특징을 확인하고, 정보 수집량과 패널티 사이에서 의사결정을 내릴 필요가 있음
- 어떤 카테고리를 대상으로 할지, 몇 개의 카테고리를 대상으로 할지 등을 결정해야 함

### 합격선과 출제 의도

- 병렬 처리를 하지 않고, 예외 처리와 배치 처리만 해도 받을 수 있는 점수인 8만 점이 합격선
- 실무에서도 흔히 발생하는 트레이드오프(의사결정 상황) 에 대한 대처 능력을 보는 것이 의도
- 평균 점수는 예외 처리만 하면 5만, 배치 처리까지 하면 14만, 병렬 처리까지 하면 25만 점임

## 3. 2018 KAKAO BLIND RECRUITMENT(3차)

> 출처 : ['카카오 신입 공채 3차 코딩 테스트 문제 해설'](https://tech.kakao.com/2017/11/14/kakao-blind-recruitment-round-3/)

- 3차는 오프라인으로 진행되었기 때문에, 개발 환경이 다르고, 검색하는 것도 불가능함
- 또한, 코딩 테스트 외에도 컴퓨팅 관련 기본 지식을 확인하는 필기시험도 진행되었음
- 이 때, 코딩 테스트는 5문제 중 3문제, 필기시험은 40점을 넘기는 것이 합격 기준임
- 코딩 테스트는 1차와 난이도가 비슷하지만, 복합적으로 생각해야 하는 문제들이 출제됨

### 3-1. N진수 게임

> 반복문과 진법 변환을 이용해 순서대로 나열된 숫자들의 특정 값만 구하기(챔퍼나운 수 활용)

> 정답률 : 91.85%

- 진법 변환은 프로그래밍 언어를 처음 배울 때 연습 문제로 많이 풀어보기 때문에 설명 없음

### 3-2. 압축

> 문자열 정보를 압축하여 색인 배열로 출력하는 LZW 알고리즘 구현

> 정답률 : 95.80%

- 압축 알고리즘이라 어렵게 생각할 수도 있지만, 설명의 의사 코드(Pseudocode) 만 구현하면 됨
- 문자열과 배열을 다루는 능력이 기초적인 수준이어도 풀 수 있으며, 정답률이 가장 높은 문제임

### 3-3. 파일명 정렬

> 대소문자 구분 없이(Case Insensitive), 숫자 값에 따라(Numerical) 안정 정렬(Stable Sort)

> 정답률 : 66.95%

- 정렬 문제에서 자주 등장하는 조건들을, 하나의 프로그램으로 작성하는 것이 까다로울 수 있음
- 알고리즘 구현 방식에 따라, 안정 정렬일 수도, 아닐 수도 있어서, 이를 확인할 필요가 있음
- 물론, 안정 정렬 알고리즘을 기본 함수로 제공하는 프로그래밍 언어도 있지만, 아닌 언어도 있음
- 비교 조건이 까다로운 경우, decorate-sort-undecorate 패턴을 이용해서 쉽게 해결할 수 있음

### 3-4. 방금그곡

> 조건에 맞게 문자열을 변환하고, 문자열 일부를 비교 (토큰화, 치환(Substitution) 활용)

> 정답률 : 47.50%

- 각 음을 나타내는 글자 수가 1일 수도, 2일 수도 있어서, 토큰화나 치환 등의 방법이 필요

### 3-5. 자동완성

> 주어진 단어들의 고유 접두어(Prefix) 찾기(트라이(Trie) 자료 구조 또는 정렬 활용)

> 정답률 : 34.07%

- 모든 단어 쌍을 비교하면 제한 시간 내에 풀 수 없어서, 트라이 자료 구조를 활용하는 것이 적합
- 전체 단어를 사전 순으로 정렬한 후에, 인접한 단어 쌍을 비교하는 방법으로도 문제 해결이 가능

### 3-6. 필기시험

> [필기시험 PDF 링크(총 20문제)](http://tech.kakao.com/files//kakao-blind-recruitment.pdf)

- 1 ~ 10번은 컴퓨팅, 컴퓨터 과학, 인터넷 등의 기본 개념의 설명을 맞추는 빈칸 채우기 문제
- 현업에서 영어로 된 문서를 자주 접하게 되므로, 독해 능력을 보기 위해 모두 영어로 구성함
- 부연 설명이 제시되며, 따로 검색하지 않고도 기본적으로 알고 있어야 할만한 용어들로 구성됨

<br>

11번 문제

- TCP 연결 종료 과정에서의 active close, passive close 를 숙지하고 있는지 묻는 문제
- 실무에서도 TIME\_WAIT 으로 인한 문제가 발생해서, 해당 개념을 숙지할 필요가 있음

<br>

12번 문제

- 제록스 팔로알토 연구소에서 1979년에 소개한, 소프트웨어 구조를 설명한 방법론에 대한 문제
- 사실상 최초의 방법론이고, 여전히 중요하게 쓰이는 MVC 소프트웨어 디자인 패턴을 묻는 문제

<br>

13 ~ 15번 문제

- 임의의 코드를 제시한 후, 시간 복잡도를 구하고, 빅오 표기법으로 표현하도록 하는 문제
- 코드 실행 순서를 파악하고, 상수항을 생략하는 등의 분석 과정에서 실수하기 쉬운 문제임

<br>

16번 문제

- 중복된 원소가 없는 리스트 A, B의 길이가 서로 다를 때, 효율적으로 비교하는 방법을 묻는 문제
- 이 때, A의 길이는 B의 길이보다 작으며, 둘 중 하나를 정렬한 후에 이진 탐색으로 비교해야 함

<br>

17 ~ 19번 문제

- 되돌리기 기능, DB 인덱스 구현, 순차적으로 메일을 보내는 체계에 적합한 자료 구조를 묻는 문제   
  `(인덱스 구현에 관한 설명은, 해시가 아닌 해시 테이블 또는 해시 맵으로 표기되는 것이 정확함)`

<br>

20번 문제

- 트리 순회 계산 문제이며, 전위/중위/후위 순회의 설명이 포함되어 모르는 사람도 쉽게 풀 수 있음

## 4. 2019 KAKAO BLIND RECRUITMENT(1차)

> 출처 : ['2019 카카오 신입 공채 1차 코딩 테스트 문제 해설'](https://tech.kakao.com/2018/09/21/kakao-blind-recruitment-for2019-round-1/)

- 작년과 비슷하게 구현 문제 위주로 구성했으며, 쉬운 난이도에서 어려운 난이도순으로 배치함
- 작년과는 다르게 효율성 테스트를 도입해, 입력 크기에 따른 효율적인 풀이를 구현하도록 유도함

### 4-1. 오픈채팅방

> 채팅방 출입, 닉네임 변경 기록이 담긴 배열을 안내 메시지가 담긴 배열로 변경(연관 배열 활용)

> 정답률 : 59.91%

- 주어진 배열을 순회하면서 메시지 종류를 기록하고, 사용자 닉네임은 연관 배열로 관리하면 됨

### 4-2. 실패율

> 주어진 사용자 정보를 기준으로 게임 스테이지별 실패율을 계산

> 정답률 : 55.57%

- 사용자가 도전 중인 스테이지의 배열을 기준으로 실패율을 계산하고, 내림차순으로 정렬하면 됨
- 주어진 배열을 순회하여 스테이지별 사용자 수를 새로운 배열에 담고, 그 배열로 실패율을 계산
- 이렇게 구한 실패율을 스테이지 번호 묶은 후, 스테이지 번호를 기준으로 내림차순 정렬하면 됨

### 4-3. 후보키

> 가능한 모든 속성의 조합을 만들고, 그중, 조건을 만족시키는 조합만 추려내기

> 정답률 : 16.09%

- dfs 또는 bit를 이용해 부분 집합을 만들고, 각 부분 집합을 이용해 중복 튜플이 있는지 검사
- 각 부분 집합을 검사한 후, 중복 튜플이 없는 키를 유일성을 만족하는 슈퍼 키 집합에 추가
- 이러한 슈퍼 키 집합에서 최소성을 만족하는 키들을 선택하여 후보 키 집합을 만들 수 있음
- 임의의 슈퍼 키 X에 대해, X의 부분 집합인 슈퍼 키 Y가 없다면, X는 후보기가 될 수 있음
- 비교적 난이도가 쉬운 편임에도 불구하고, 부분 집합을 만드는 부분 때문에 어려운 편이었음

### 4-4. 무지의 먹방 라이브

> 여러 음식을 1초에 한 입씩, 순서대로 먹는 상황에서, 특정 시점에 먹게 되는 음식 알아내기 

> 정답률 : 정확성 42.08% / 효율성 5.52%

- 반복문을 이용해, 1초가 지날 때마다 다음 먹을 음식을 하나씩 찾아가며 시뮬레이션하면 됨
- 이렇게 하나씩 처리하면 효율성 테스트를 통과할 수 없어서, 시뮬레이션의 효율을 높여야 함
- 음식별로 필요 시간을 정렬하고, 각 시간에서 가장 작은 필요 시간의 값을 빼는 방법도 있음
- 이 때, 건너뛸 수 있는 시간은 (남은 필요 시간 중 가장 작은 값) * (남은 음식의 수) 만큼임
- '음식 하나를 다 먹은 시점' 으로 건너뛰게 되어서, 음식 단위로 시뮬레이션을 진행할 수 있음
- 이외에도 다양한 방법이 있으며, 주어진 시점에 빠르게 도달하는 방법을 찾는 것이 핵심임

### 4-5. 길 찾기 게임

> 주어진 좌표들을 노드로 하는 트리를 구성하고, 전위/후위 순회 경로를 알아내기

> 정답률 : 7.40%

- 트리를 순회하는 방법에 대해서는 검색을 통해 쉽게 알아낼 수 있어서, 크게 문제가 되지 않음
- 따라서, 해당 문제의 핵심은 입력으로 주어지는 좌표 값들을 기준으로 트리를 구성하는 부분임
- y 값을 기준으로 트리의 계층을 구분하고, x 값을 기준으로 부모/자식 관계를 찾는 것이 중요함
- 임의의 노드 a와 a의 왼쪽 자식 노드 b가 있을 때, b의 자식 노드들은, a보다 왼쪽에 있어야 함
- a의 오른쪽 자식 노드를 c라고 하면, 마찬가지로, c의 자식 노드들은 a보다 오른쪽에 있어야 함
- 이를 재귀적으로 반복하여 트리를 구성할 수 있으며, 이외에도 자식을 찾는 여러 방법이 있음

### 4-6. 매칭 점수

> 점수 규칙에 따라, 웹 페이지의 매칭 점수 계산하기 (백링크를 포함하는 검색 알고리즘 구현)

> 정답률 : 6.12%

- 점수 계산 로직은 단순하지만, 점수 계산에 필요한 요소들을 추출하는 것이 어려운 문제임
- 정규 표현식이나 문자열 처리 로직을 이용해, 웹 페이지와 외부 링크의 URL을 찾아내야 함
- 기본 점수를 제대로 계산하기 위해선, 포함하는 단어가 아닌, 일치하는 단어를 찾아야 함

### 4-7. 블록 게임

> 배치된 블록 중, 몇 개의 블록을 제거할 수 있는지 알아내기

> 정답률 : 5.85%

- 문제 자체를 이해하기는 쉽지만, 제거할 수 있는 블록을 확인하기 위한 아이디어가 필요함
- 조건에 따라, 채울 수 있는 빈칸을 모두 채운 후, 조건을 충족한 블록들을 제거해도 됨
- 이외에도, 제거할 수 있는 블록의 모양과 조건을 검사하여 블록을 제거하는 방법도 있음

## 5. 2019 KAKAO BLIND RECRUITMENT(2차)

> 출처 : ['2019 카카오 블라인드 공채 2차 오프라인 코딩 테스트 문제 해설'](https://tech.kakao.com/2018/10/23/kakao-blind-recruitment-round-2/)

- 작년 시험에 비해 시험 시간이 줄어들었으며, 체계 설계 역량을 평가하는 데에 더 집중함

> 문제 내용 : 효율적인 엘리베이터 알고리즘 구현

### 문제 조건 설명

- 엘리베이터 동작/상태 관리 서버가 제공되며, 이러한 정보들은 REST API, JSON으로 통신
- 지원자는 빌딩별 승객 트래픽 정보를 분석한 후, 엘리베이터 제어 알고리즘을 구현해야 함
- 엘리베이터는 0부터 시작하여, 명령마다 1씩 증가하는 가상의 시간(timestamp) 을 사용함
- 승객이 보내는 탑승 요청(call) 에는 탑승 위치와 목적지 층에 대한 정보가 포함되어 있음
- 이 때, 승객들은 독립적인 행동을 하지 않으며, 엘리베이터 시스템의 지시에 따라 움직임
- 또, 목적지가 아닌 다른 층에 내려지는 승객은, 목적지로 가기 위해 다시 탑승 요청을 보냄
- 사용할 수 있는 엘리베이터는 여러 대이며, 주어진 엘리베이터를 모두 사용하지 않아도 됨
- 엘리베이터에 명령을 내리면, 이동, 멈추기, 문 여닫기, 승객 태우기/내리기를 할 수 있음
- 엘리베이터에는 정원이 있으며, 멈춤, 열림, 위로, 아래로 등의 상태 표시값(status) 이 있음
- 멈추기, 위로, 아래로, 열기, 닫기, 태우기, 내리기, 이렇게 총 7개의 명령을 사용할 수 있음

### 문제의 특징 및 의도

- 실생활의 엘리베이터 알고리즘은 'collective control', 'elevator algorithm' 등으로 불림
- 탑승객이 있거나, 진행 방향과 같은 방향으로 이동하는 승객이 있으면 방향을 유지해야 함
- 현재 진행 방향의 요청이 모두 처리되면 방향을 전환하며, 요청이 없다면 그대로 멈추면 됨
- 위의 두 가지 규칙을 따르는 아주 간단한 알고리즘이며, 디스크 스케줄링과 유사한 점이 있음
- 이렇게, 엘리베이터 1대만을 관리하는 상황과는 다르게, 해당 문제의 조건은 조금 더 복잡함
- 승객의 요청을 어떤 엘리베이터에 할당할지를 결정하는 것만으로도, 문제가 아주 복잡해짐
- 요청을 더 효율적으로 처리하기 위해 비용 계산이 필요하며, 승객의 위치도 관리해야 함
- 엘리베이터마다 구역을 나눠서 운영하는 경우(홀/짝, 고층/저층), 환승 기능도 구현해야 함
- 문제마다 승객 패턴이 다르며, 모든 문제를 풀기 위해선 다수의 엘리베이터를 운영해야 함
- 여러 상황에 대비하기 위해서는, 추상화와 모듈화를 통해 체계 구현의 유연성을 높여야 함

### 승객 트래픽 모델링

- 엘리베이터 요청은 1층에서 특정 층, 특정 층에서 1층, 1층이 아닌 다른 층간의 이동이 있음
- 문제는 총 3개, 모두 다른 건물이 제시되며, 전체 층수, 요청 수, 요청 패턴 등이 모두 다름

### 다양한 접근 방법

- 'FIFO' 방식은 가장 쉬운 접근 방법이지만, 1번 이외의 문제에서는 좋은 점수를 받을 수 없음
- 'Collective Control' 알고리즘을 이용하면, 엘리베이터를 1대만으로도 효율적인 운영이 가능
- 지원자 중 일부는, 더 좋은 점수를 얻기 위해, timestamp 값이 추가되는 행위들을 최소화함
- 이를 위해, 요청이 쌓일 때까지 기다리거나, 승객을 최대한으로 유지하는 등의 방식이 사용됨
