## 1. 2018 KAKAO BLIND RECRUITMENT(1차)

> 출처 : ['카카오 신입 공채 1차 코딩 테스트 문제 해설'](https://tech.kakao.com/2017/09/27/kakao-blind-recruitment-round-1/)

- 업무에서 있을 만한 상황을 가정, 다양한 분야에서 출제한 비교적 쉬운 수준의 독창적인 문제
- 자료 구조, 알고리즘 등의 전산학 기초를 충분히 학습한 사람은 누구나 풀 수 있을 만한 문제

### 1-1. 비밀 지도

> 비트 연산(Bitwise Operation)

> 정답률 : 81.78%

- 예시에 2진수 처리가 필요함을 암시하는 힌트가 있고, 둘 중 하나가 1인 경우를 처리하면 됨
- if else 구문을 이용해 푸는 방법도 있지만, 비트 연산 활용 능력을 확인하는 것이 문제의 의도

### 1-2. 다트 게임

> 문자열 처리(String Manipulation), 토큰화(Tokenizing) 와 의미 분석(Semantic Analysis)

> 정답률 : 73.47%

- 앞에서부터 한 글자씩 잘라서 처리하거나, 컴파일러처럼 토큰화와 의미 분석을 통해 계산 가능
- 정규식으로 비교적 쉽게 풀 수 있으며, 두 자리 숫자와 음수 점수가 있다는 점에 유의해야 함

### 1-3. 캐시

> LRU 캐시 교체 알고리즘 구현(검색 가능. 단, 출처 명시는 필수)

> 정답률 : 45.26%

- 입출력에 캐시 사이즈 0이 포함되어 있어서, 단순한 코드 복사/붙여넣기만으로는 해결 불가
- 입출력 예제 결과를 분석하면 쉽게 풀 수 있으며, 비정상적인 상황에 대한 예외 처리가 필요

### 1-4. 셔틀버스

> 문자열로 주어지는 시간을 조건에 맞게 계산

> 정답률 : 26.79%

- 일부 조건이 까다로운 편이며, 전체적인 계산은 어렵지 않지만, 정확히 계산하는 것이 어려움

### 1-5. 뉴스 클러스터링

> 자카드 유사도를 계산하는 프로그램 작성(다중집합(multiset) or 병합 정렬(Merge Sort) 활용)

> 정답률 : 41.84%

- 기본적으로, 문제 풀이에 필요한 요소인 자카드 유사도와 다중집합에 대한 설명이 주어짐
- 일부 언어에서 기본적으로 제공하지 않고, 자주 접하는 자료 구조가 아닌 다중집합을 활용
- 다중집합 자료 구조를 사용하지 않아도, 정렬된 배열과 병합 정렬을 응용하는 풀이법도 있음
- 교집합과 합집합을 잘 구하고, 공집합 나눗셈(division by zero) 에 유의하면 쉽게 풀 수 있음

### 1-6. 프렌즈4블록

> 게임 요구 사항을 구현

> 정답률 : 48.01%

- 블록 매트릭스를 생성하여 스캔하고, 조건에 맞는 블록 덩어리를 제거하는 작업을 반복해야 함
- 제거할 블록이 남아있지 않을 때까지 작업을 반복한 후, 사라진 블록 자리의 수를 계산하면 됨

### 1-7. 추석 트래픽

> 초당 최대 처리량이 되는 구간 윈도우 찾기

> 정답률 : 17.99%

- 전부 스캔하기에는 범위가 너무 넓고, 단위 또한 작아서, 필요한 연산의 수가 너무 많아짐
- 타임아웃이 발생하는 것을 막기 위해, 스캔 범위를 줄일 방법을 도출해내는 것이 중요함
- 이러한 방법은, 각 로그의 시작과 끝에서 요청량이 변한다는 규칙을 통해 알아낼 수 있음
- 이 방법 외에도, 윈도우를 이용하지 않고, O(n log n) 의 시간 복잡도로 푸는 방법도 있음

## 2. 2018 KAKAO BLIND RECRUITMENT(2차)

> 출처 : ['카카오 신입 공채 2차 코딩 테스트 문제 해설'](https://tech.kakao.com/2017/10/24/kakao-blind-recruitment-round-2-2/)

- 1차에서는 자료 구조, 알고리즘 등 전산학 기초 지식과 기본적인 구현 역량을 확인
- 2차는 빠른 습득력, 요구사항 분석과 기능 설계 능력, 모니터링/개선 능력을 확인

> 문제 내용 : 이미지 특징값을 수집하는 Crawler 구현

### REST API와 JSON Parsing

- 우선, 토큰을 발급받은 후에, 테스트 전용 서버로 연결되는 5개 카테고리의 URL 을 받아야 함
- 카테고리의 문서를 읽고, 그 안에 있는 이미지들의 특징값을 추출하여 저장/삭제하는 문제임

### 반복적인 모니터링과 최적화(제한된 토큰 유효시간)

- 토큰 유효기간은 10분으로 설정, 점수 현황판을 공개하여 지원자의 최적화 작업을 유도함
- 문제가 발생할 때마다 로그를 남기고, 모니터링을 통해 원인이나 병목 현상을 찾도록 유도함

### 평가 메트릭 분석 후에 전략 수립

- 누락되거나 삭제되지 않은 이미지, 잘못된 정보의 수에는 패널티가 붙도록 기준을 정함
- 실제 상황에서 얼마나 큰 문제가 될 수 있는지에 따라서 패널티의 크기를 다르게 적용함
- 참가자들이 불필요한 요청의 수를 줄이도록, 전체 쿼리의 양에도 약간의 패널티를 부여함

### 고득점을 위한 병렬 처리

- 가장 자주 호출하게 되는 이미지 엔드포인트의 처리 속도를 조절하여 병렬 처리를 유도함
- 처리 속도는 16ms ~ 120ms 이므로, 네트워크 지연을 포함해 초당 요청 제한은 50건 정도
- 이 때, 테스트 전용 서버의 처리 속도는 배치 처리 개수에 따라서 exp 그래프 형태를 띰
- 따라서, 최대 배치 처리 사이즈인 50개보다, 30 ~ 40개 정보의 요청에서 가장 효율이 좋음

### 예외 처리

- 크롤링 작업 중에 사용자가 이미지를 삭제하거나 서버가 다운되는 경우도 반영되어 있음
- 따라서, 확률적으로 성공하는지, 무조건 실패하는지에 따라 재시도나 블록 처리가 필요함

### 다양한 시나리오에 대응

- 중복 추가, 추가 후 바로 삭제, 없는 이미지 삭제, 추가 삭제 추가가 반복되는 상황도 있음
- 이 때, 모든 이미지를 색인하면서 발생하는 공간 낭비는 간단한 메모리 캐시로도 해결 가능
- 연산 목록은 단순히 set 자료형을 활용해 중복을 상쇄하면 문제가 발생하도록 구성되어 있음
- 또한, 추가/삭제 연산이 발생하는 범위가 넓어서, 시간과 정확성 사이에서 의사결정이 필요

### 카테고리별 특성 파악

- 카테고리마다 추가/삭제 비율과 URL 갱신 시간에 차이가 있어서, 적절한 전략이 필요함
- 카테고리별 특징을 확인하고, 정보 수집량과 패널티 사이에서 의사결정을 내릴 필요가 있음
- 어떤 카테고리를 대상으로 할지, 몇 개의 카테고리를 대상으로 할지 등을 결정해야 함

### 합격선과 출제 의도

- 병렬 처리를 하지 않고, 예외 처리와 배치 처리만 해도 받을 수 있는 점수인 8만 점이 합격선
- 실무에서도 흔히 발생하는 트레이드오프(의사결정 상황) 에 대한 대처 능력을 보는 것이 의도
- 평균 점수는 예외 처리만 하면 5만, 배치 처리까지 하면 14만, 병렬 처리까지 하면 25만 점임

## 3. 2018 KAKAO BLIND RECRUITMENT(3차)

> 출처 : ['카카오 신입 공채 3차 코딩 테스트 문제 해설'](https://tech.kakao.com/2017/11/14/kakao-blind-recruitment-round-3/)

- 3차는 오프라인으로 진행되었기 때문에, 개발 환경이 다르고, 검색하는 것도 불가능함
- 또한, 코딩 테스트 외에도 컴퓨팅 관련 기본 지식을 확인하는 필기시험도 진행되었음
- 이 때, 코딩 테스트는 5문제 중 3문제, 필기시험은 40점을 넘기는 것이 합격 기준임
- 코딩 테스트는 1차와 난이도가 비슷하지만, 복합적으로 생각해야 하는 문제들이 출제됨

### 3-1. N진수 게임

> 반복문과 진법 변환을 이용해 순서대로 나열된 숫자들의 특정 값만 구하기(챔퍼나운 수 활용)

> 정답률 : 91.85%

- 진법 변환은 프로그래밍 언어를 처음 배울 때 연습 문제로 많이 풀어보기 때문에 설명 없음

### 3-2. 압축

> 문자열 정보를 압축하여 색인 배열로 출력하는 LZW 알고리즘 구현

> 정답률 : 95.80%

- 압축 알고리즘이라 어렵게 생각할 수도 있지만, 설명의 의사 코드(Pseudocode) 만 구현하면 됨
- 문자열과 배열을 다루는 능력이 기초적인 수준이어도 풀 수 있으며, 정답률이 가장 높은 문제임

### 3-3. 파일명 정렬

> 대소문자 구분 없이(Case Insensitive), 숫자 값에 따라(Numerical) 안정 정렬(Stable Sort)

> 정답률 : 66.95%

- 정렬 문제에서 자주 등장하는 조건들을, 하나의 프로그램으로 작성하는 것이 까다로울 수 있음
- 알고리즘 구현 방식에 따라, 안정 정렬일 수도, 아닐 수도 있어서, 이를 확인할 필요가 있음
- 물론, 안정 정렬 알고리즘을 기본 함수로 제공하는 프로그래밍 언어도 있지만, 아닌 언어도 있음
- 비교 조건이 까다로운 경우, decorate-sort-undecorate 패턴을 이용해서 쉽게 해결할 수 있음

### 3-4. 방금그곡

> 조건에 맞게 문자열을 변환하고, 문자열 일부를 비교(토큰화, 치환(Substitution) 활용)

> 정답률 : 47.50%

- 각 음을 나타내는 글자 수가 1일 수도, 2일 수도 있어서, 토큰화나 치환 등의 방법이 필요

### 3-5. 자동완성

> 주어진 단어들의 고유 접두어(Prefix) 찾기(트라이(Trie) 자료 구조 또는 정렬 활용)

> 정답률 : 34.07%

- 모든 단어 쌍을 비교하면 제한 시간 내에 풀 수 없어서, 트라이 자료 구조를 활용하는 것이 적합
- 전체 단어를 사전 순으로 정렬한 후에, 인접한 단어 쌍을 비교하는 방법으로도 문제 해결이 가능

### 3-6. 필기시험

> [필기시험 PDF 링크(총 20문제)](http://tech.kakao.com/files//kakao-blind-recruitment.pdf)

- 1 ~ 10번은 컴퓨팅, 컴퓨터 과학, 인터넷 등의 기본 개념의 설명을 맞추는 빈칸 채우기 문제
- 현업에서 영어로 된 문서를 자주 접하게 되므로, 독해 능력을 보기 위해 모두 영어로 구성함
- 부연 설명이 제시되며, 따로 검색하지 않고도 기본적으로 알고 있어야 할만한 용어들로 구성됨

<br>

11번 문제

- TCP 연결 종료 과정에서의 active close, passive close 를 숙지하고 있는지 묻는 문제
- 실무에서도 TIME\_WAIT 으로 인한 문제가 발생해서, 해당 개념을 숙지할 필요가 있음

<br>

12번 문제

- 제록스 팔로알토 연구소에서 1979년에 소개한, 소프트웨어 구조를 설명한 방법론에 대한 문제
- 사실상 최초의 방법론이고, 여전히 중요하게 쓰이는 MVC 소프트웨어 디자인 패턴을 묻는 문제

<br>

13 ~ 15번 문제

- 임의의 코드를 제시한 후, 시간 복잡도를 구하고, 빅오 표기법으로 표현하도록 하는 문제
- 코드 실행 순서를 파악하고, 상수항을 생략하는 등의 분석 과정에서 실수하기 쉬운 문제임

<br>

16번 문제

- 중복된 원소가 없는 리스트 A, B의 길이가 서로 다를 때, 효율적으로 비교하는 방법을 묻는 문제
- 이 때, A의 길이는 B의 길이보다 작으며, 둘 중 하나를 정렬한 후에 이진 탐색으로 비교해야 함

<br>

17 ~ 19번 문제

- 되돌리기 기능, DB 인덱스 구현, 순차적으로 메일을 보내는 체계에 적합한 자료 구조를 묻는 문제   
  `(인덱스 구현에 관한 설명은, 해시가 아닌 해시 테이블 또는 해시 맵으로 표기되는 것이 정확함)`

<br>

20번 문제

- 트리 순회 계산 문제이며, 전위/중위/후위 순회의 설명이 포함되어 모르는 사람도 쉽게 풀 수 있음

## 4. 2019 KAKAO BLIND RECRUITMENT(1차)

> 출처 : ['2019 카카오 신입 공채 1차 코딩 테스트 문제 해설'](https://tech.kakao.com/2018/09/21/kakao-blind-recruitment-for2019-round-1/)

- 작년과 비슷하게 구현 문제 위주로 구성했으며, 쉬운 난이도에서 어려운 난이도순으로 배치함
- 작년과는 다르게 효율성 테스트를 도입해, 입력 크기에 따른 효율적인 풀이를 구현하도록 유도함

### 4-1. 오픈채팅방

> 채팅방 출입, 닉네임 변경 기록이 담긴 배열을 안내 메시지가 담긴 배열로 변경(연관 배열 활용)

> 정답률 : 59.91%

- 주어진 배열을 순회하면서 메시지 종류를 기록하고, 사용자 닉네임은 연관 배열로 관리하면 됨

### 4-2. 실패율

> 주어진 사용자 정보를 기준으로 게임 스테이지별 실패율을 계산

> 정답률 : 55.57%

- 사용자가 도전 중인 스테이지의 배열을 기준으로 실패율을 계산하고, 내림차순으로 정렬하면 됨
- 주어진 배열을 순회하여 스테이지별 사용자 수를 새로운 배열에 담고, 그 배열로 실패율을 계산
- 이렇게 구한 실패율을 스테이지 번호 묶은 후, 스테이지 번호를 기준으로 내림차순 정렬하면 됨

### 4-3. 후보키

> 가능한 모든 속성의 조합을 만들고, 그중, 조건을 만족시키는 조합만 추려내기

> 정답률 : 16.09%

- dfs 또는 bit를 이용해 부분 집합을 만들고, 각 부분 집합을 이용해 중복 튜플이 있는지 검사
- 각 부분 집합을 검사한 후, 중복 튜플이 없는 키를 유일성을 만족하는 슈퍼 키 집합에 추가
- 이러한 슈퍼 키 집합에서 최소성을 만족하는 키들을 선택하여 후보 키 집합을 만들 수 있음
- 임의의 슈퍼 키 X에 대해, X의 부분 집합인 슈퍼 키 Y가 없다면, X는 후보기가 될 수 있음
- 비교적 난이도가 쉬운 편임에도 불구하고, 부분 집합을 만드는 부분 때문에 어려운 편이었음

### 4-4. 무지의 먹방 라이브

> 여러 음식을 1초에 한 입씩, 순서대로 먹는 상황에서, 특정 시점에 먹게 되는 음식 알아내기 

> 정답률 : 정확성 42.08% / 효율성 5.52%

- 반복문을 이용해, 1초가 지날 때마다 다음 먹을 음식을 하나씩 찾아가며 시뮬레이션하면 됨
- 이렇게 하나씩 처리하면 효율성 테스트를 통과할 수 없어서, 시뮬레이션의 효율을 높여야 함
- 음식별로 필요 시간을 정렬하고, 각 시간에서 가장 작은 필요 시간의 값을 빼는 방법도 있음
- 이 때, 건너뛸 수 있는 시간은 (남은 필요 시간 중 가장 작은 값) * (남은 음식의 수) 만큼임
- '음식 하나를 다 먹은 시점' 으로 건너뛰게 되어서, 음식 단위로 시뮬레이션을 진행할 수 있음
- 이외에도 다양한 방법이 있으며, 주어진 시점에 빠르게 도달하는 방법을 찾는 것이 핵심임

### 4-5. 길 찾기 게임

> 주어진 좌표들을 노드로 하는 트리를 구성하고, 전위/후위 순회 경로를 알아내기

> 정답률 : 7.40%

- 트리를 순회하는 방법에 대해서는 검색을 통해 쉽게 알아낼 수 있어서, 크게 문제가 되지 않음
- 따라서, 해당 문제의 핵심은 입력으로 주어지는 좌표 값들을 기준으로 트리를 구성하는 부분임
- y 값을 기준으로 트리의 계층을 구분하고, x 값을 기준으로 부모/자식 관계를 찾는 것이 중요함
- 임의의 노드 a와 a의 왼쪽 자식 노드 b가 있을 때, b의 자식 노드들은, a보다 왼쪽에 있어야 함
- a의 오른쪽 자식 노드를 c라고 하면, 마찬가지로, c의 자식 노드들은 a보다 오른쪽에 있어야 함
- 이를 재귀적으로 반복하여 트리를 구성할 수 있으며, 이외에도 자식을 찾는 여러 방법이 있음

### 4-6. 매칭 점수

> 점수 규칙에 따라, 웹 페이지의 매칭 점수 계산하기(백링크를 포함하는 검색 알고리즘 구현)

> 정답률 : 6.12%

- 점수 계산 로직은 단순하지만, 점수 계산에 필요한 요소들을 추출하는 것이 어려운 문제임
- 정규 표현식이나 문자열 처리 로직을 이용해, 웹 페이지와 외부 링크의 URL을 찾아내야 함
- 기본 점수를 제대로 계산하기 위해선, 포함하는 단어가 아닌, 일치하는 단어를 찾아야 함

### 4-7. 블록 게임

> 배치된 블록 중, 몇 개의 블록을 제거할 수 있는지 알아내기

> 정답률 : 5.85%

- 문제 자체를 이해하기는 쉽지만, 제거할 수 있는 블록을 확인하기 위한 아이디어가 필요함
- 조건에 따라, 채울 수 있는 빈칸을 모두 채운 후, 조건을 충족한 블록들을 제거해도 됨
- 이외에도, 제거할 수 있는 블록의 모양과 조건을 검사하여 블록을 제거하는 방법도 있음

## 5. 2019 KAKAO BLIND RECRUITMENT(2차)

> 출처 : ['2019 카카오 블라인드 공채 2차 오프라인 코딩 테스트 문제 해설'](https://tech.kakao.com/2018/10/23/kakao-blind-recruitment-round-2/)

- 작년 시험에 비해 시험 시간이 줄어들었으며, 체계 설계 역량을 평가하는 데에 더 집중함

> 문제 내용 : 효율적인 엘리베이터 알고리즘 구현

### 문제 조건 설명

- 엘리베이터 동작/상태 관리 서버가 제공되며, 이러한 정보들은 REST API, JSON으로 통신
- 지원자는 빌딩별 승객 트래픽 정보를 분석한 후, 엘리베이터 제어 알고리즘을 구현해야 함
- 엘리베이터는 0부터 시작하여, 명령마다 1씩 증가하는 가상의 시간(timestamp) 을 사용함
- 승객이 보내는 탑승 요청(call) 에는 탑승 위치와 목적지 층에 대한 정보가 포함되어 있음
- 이 때, 승객들은 독립적인 행동을 하지 않으며, 엘리베이터 시스템의 지시에 따라 움직임
- 또, 목적지가 아닌 다른 층에 내려지는 승객은, 목적지로 가기 위해 다시 탑승 요청을 보냄
- 사용할 수 있는 엘리베이터는 여러 대이며, 주어진 엘리베이터를 모두 사용하지 않아도 됨
- 엘리베이터에 명령을 내리면, 이동, 멈추기, 문 여닫기, 승객 태우기/내리기를 할 수 있음
- 엘리베이터에는 정원이 있으며, 멈춤, 열림, 위로, 아래로 등의 상태 표시값(status) 이 있음
- 멈추기, 위로, 아래로, 열기, 닫기, 태우기, 내리기, 이렇게 총 7개의 명령을 사용할 수 있음

### 문제의 특징 및 의도

- 실생활의 엘리베이터 알고리즘은 'collective control', 'elevator algorithm' 등으로 불림
- 탑승객이 있거나, 진행 방향과 같은 방향으로 이동하는 승객이 있으면 방향을 유지해야 함
- 현재 진행 방향의 요청이 모두 처리되면 방향을 전환하며, 요청이 없다면 그대로 멈추면 됨
- 위의 두 가지 규칙을 따르는 아주 간단한 알고리즘이며, 디스크 스케줄링과 유사한 점이 있음
- 이렇게, 엘리베이터 1대만을 관리하는 상황과는 다르게, 해당 문제의 조건은 조금 더 복잡함
- 승객의 요청을 어떤 엘리베이터에 할당할지를 결정하는 것만으로도, 문제가 아주 복잡해짐
- 요청을 더 효율적으로 처리하기 위해 비용 계산이 필요하며, 승객의 위치도 관리해야 함
- 엘리베이터마다 구역을 나눠서 운영하는 경우(홀/짝, 고층/저층), 환승 기능도 구현해야 함
- 문제마다 승객 패턴이 다르며, 모든 문제를 풀기 위해선 다수의 엘리베이터를 운영해야 함
- 여러 상황에 대비하기 위해서는, 추상화와 모듈화를 통해 체계 구현의 유연성을 높여야 함

### 승객 트래픽 모델링

- 엘리베이터 요청은 1층에서 특정 층, 특정 층에서 1층, 1층이 아닌 다른 층간의 이동이 있음
- 문제는 총 3개, 모두 다른 건물이 제시되며, 전체 층수, 요청 수, 요청 패턴 등이 모두 다름

### 다양한 접근 방법

- 'FIFO' 방식은 가장 쉬운 접근 방법이지만, 1번 이외의 문제에서는 좋은 점수를 받을 수 없음
- 'Collective Control' 알고리즘을 이용하면, 엘리베이터를 1대만으로도 효율적인 운영이 가능
- 지원자 중 일부는, 더 좋은 점수를 얻기 위해, timestamp 값이 추가되는 행위들을 최소화함
- 이를 위해, 요청이 쌓일 때까지 기다리거나, 승객을 최대한으로 유지하는 등의 방식이 사용됨

## 6. 2020 KAKAO BLIND RECRUITMENT(1차)

> 출처 : ['2020 신입 개발자 블라인드 채용 1차 코딩 테스트 문제 해설'](https://tech.kakao.com/2019/10/02/kakao-blind-recruitment-2020-round1/)

- 모든 테스트 케이스를 통과해야만 정답 처리, 효율성 테스트가 있는 4번 문제는 부분 점수 적용

### 6-1. 문자열 압축

> 특정 길이를 기준으로 하여, 문자열을 연속 길이 부호화(LRE) 방식으로 압축하기

> 정답률 : 25.85%

- 문자열 자르기, 비교하기, 길이 얻기, 부분 문자열 얻기 등의 작업을 할 줄 아는지 확인하는 문제
- 입력으로 주어지는 문자열의 최대 길이가 1,000이므로, 가능한 모든 압축 형태를 시도해봐도 됨
- 주어진 문자열의 길이가 n일 때, 압축에 사용될 수 있는 부분 문자열의 최대 길이는 (n / 2) 임

### 6-2. 괄호 변환

> 모든 괄호의 짝을 맞추는 알고리즘 작성하기(주어진 로직 구현, 재귀 함수 작성)

> 정답률 : 23.1%

- 짝이 맞지 않는 괄호들로 구성된 문자열이 주어졌을 때, 해당 문자열을 변환하는 방법이 제시됨
- 괄호 쌍 확인법과 재귀 함수를 이해하고 있고, 주어진 로직대로 코딩할 줄 알면 쉽게 풀 수 있음

### 6-3. 자물쇠와 열쇠

> 2차원 배열로 표현된 자물쇠와 열쇠에 대해, 자물쇠의 빈칸을 열쇠로 메꿀 수 있는지 알아내기

> 정답률 : 7.4%

- 2차원 배열 활용 능력을 확인하는 문제이며, 입력 크기가 작아서 가능한 모든 경우를 확인 가능
- 주어진 배열을 회전시키면서 가능한 모든 경우를 확인, 조건에 맞는 결과가 있는지를 반환하면 됨
- 자물쇠와 열쇠를 배치할 수 있는 새로운 배열을 만들고, 모든 조합을 시뮬레이션 하면 풀 수 있음
- 2차원 배열을 회전시키는 함수를 따로 구현해두면, 정답 코드를 좀 더 깔끔하게 작성할 수 있음

### 6-4. 가사 검색

> 주어진 규칙에 따라 문자열 비교하기(트라이 자료 구조 또는 이분 탐색 활용)

> 정답률 : 정확성 34.4% / 효율성 0.8%

- 문자열을 다룰 수 있는지 확인하는 문제이며, 효율적인 방법을 사용할 줄 알아야 풀 수 있는 문제
- 검색 기준이 되는 queries 배열과 비교 대상인 words 배열의 원소들을 하나씩 비교하면 풀 수 있음
- 효율성 테스트를 통과하려면, 비교 대상 트라이를 만들거나, 비교 대상 정렬 후 이분 탐색을 해야 함
- 검색 기준을 모두 처리하려면, 일반 문자열과 뒤집힌 문자열에 대한 트라이를 하나씩 만들어야 함
- 검색 기준 문자열의 길이에 따라 다른 트라이를 탐색하도록, 길이를 기준으로 트라이를 생성해야 함
- 트라이를 생성할 때, 철자 노드를 만날 때마다 횟수를 갱신하면, 탐색 결과를 빠르게 얻을 수 있음

### 6-5. 기둥과 보 설치

> 조건에 맞게 2차원 배열의 값 변경하기

> 정답률 : 1.9%

- '기둥과 보의 설치/삭제 조건을 정확하게 따르면서, 입력을 시뮬레이션하는 코드' 를 작성하는 문제

### 6-6. 외벽 점검

> 원 형태의 정보가 담긴 배열에서, 특정 지점들을, 주어진 범위만 사용하여 완전 탐색하기

> 정답률 : 0.6%

- bit mask 또는 permutation 등을 활용하여, 원형으로 주어진 완전 탐색 문제를 풀 수 있어야 함
- 확인해야 하는 탐색 범위의 수는 최대 8개이므로, 가능한 모든 경우를 확인하여 문제를 풀 수 있음
- 원 형태의 정보는 직선 정보로 볼 수도 있으므로, 탐색 구간은 시작 지점부터 순서대로 배정해야 함
- 특정 구간을 탐색했음에도, 탐색해야 하는 지점이 남아있는 경우, 그다음 지점부터 다시 탐색하면 됨
- 주어진 범위만으로 완전 탐색이 가능한지와 주어진 범위를 최소한으로 사용하는 경우를 확인해야 함

### 6-7. 블록 이동하기

> 2차원 평면에 있는 '이동/회전이 가능한 로봇' 을 목적지까지 옮기는 데 필요한 최소 움직임 수 구하기

> 정답률 : 1.4%

- 너비 우선 탐색을 단순히 외워서 알고 있는지, 응용하여 코드를 작성할 수 있는지 확인하는 문제임
- 로봇은 2차원 평면 격자를 기준으로 1 * 2 크기이며, 장애물이 없다면 회전하고 이동할 수 있음
- 상하좌우 이동이 가능하며, 두 칸 중 하나를 축으로 하여 시계/반 시계 방향으로 회전할 수 있음
- 충돌 판정을 확인하여 이동 가능 여부를 판별하고, 너비 우선 탐색을 진행하면 문제를 풀 수 있음

> 출처 : ['2020 카카오 블라인드 공채 2차 오프라인 코딩 테스트 문제 해설'](https://tech.kakao.com/2019/10/21/kakao-blind-recruitment-2020-round2/)

> 문제 내용 : 가상의 SNS 카카오팔로잉을 위한 팔로잉 추천 시스템 구현

### 카카오팔로잉

- 카카오팔로잉은 팔로우 중인 사용자의 글과 사진을 볼 수 있는 팔로우 기반 구독형 SNS 서비스임
- 전화번호가 ID 역할을 하며, 사용자의 전화번호부는 서버에 등록, 변경될 때마다 서버에 동기화됨
- 처음 가입한 사용자는 어떤 사용자도 팔로우하지 않은 상태이며, 팔로우는 상대방 동의 없이 가능
- 이 때, 사용자는 팔로잉 추천 시스템에서 제공하는 추천 목록에서만 팔로우 상대를 고를 수 있음

### 팔로잉 추천 시스템 구현

- 모든 사용자가 20명 이상의 다른 사용자를 팔로우하도록 하는 것이 팔로잉 추천 시스템의 목표임
- 추천 시스템은 서버와 JSON 정보를 주고받는 REST API이며, 하루에 한 번씩 추천 작업을 수행함
- 추천 로직에 필요한 정보 다운로드, 추천 로직 수행, 추천 목록 업로드의 순서로 작업을 수행함

### 추천 시뮬레이션

- 팔로잉 추천 시스템의 성능은 시뮬레이션을 통해 평가되며, 시뮬레이션은 1일 단위로 진행됨
- 시뮬레이션은, 진행일을 기준으로 추천 작업에 필요한 모든 내용이 준비된 상태에서 진행됨
- 시뮬레이션은, 진행일에 업로드된 추천 목록을 기준으로 하며 목표가 달성될 때까지 진행됨

### 시뮬레이션 시나리오

1. 최초 가입자 100명, 추가 가입 및 전화번호부 변경은 없음
2. 최초 가입자 3,000명, 이후 30일 동안 매일 신규 가입 및 전화번호부 변경 이벤트 있음

> 공통 제약사항 : 진행일 기준 전체 사용자 수만큼, 사용자마다 최대 10명까지 추천 가능

### 팔로우 확률

- 시뮬레이션에서 사용자가 다른 사용자를 팔로우할 확률은 사용자 간의 관계에 따라 달라짐  
  `(아래에서 '특정 사용자' 는 시뮬레이션 대상인 사용자, '추천 대상'은 추천 목록에 있는 사용자임)`
- 추천 대상이 특정 사용자의 전화번호부에 없는 사용자면 5%p, 있는 사용자면 30%p 증가함
- 추천 대상을 팔로우하는 사용자가 특정 사용자의 전화번호부에 있다면, 1명당 10%p 증가함
- 추천 대상을 팔로우하는 사용자를 특정 사용자가 팔로우하고 있는 경우, 1명당 10%p 증가함

### 출제 의도

성능이 우수한 추천 시스템은 최소한의 추천 횟수로 목표를 달성할 수 있는 추천 시스템임

- 따라서, 지원자는 추천 횟수를 최소화하기 위해 적합한 추천 방법을 선택할 필요가 있음
- 하지만, 알고리즘 활용 능력이 아닌, 종합적인 문제 해결 능력을 평가하는 것이 목표임
- 때문에, 평가 기준은 목표를 달성하는 것만으로도 적지 않은 점수를 받도록 조정되어 있음

<br>

시뮬레이션 시나리오는 난이도와 문제 해결 단계에 따라 1과 2로 나뉘어서 구성되어 있음

- 1번 시나리오에서는 API 연동으로 문제에 익숙해질 수 있고, 문제 이해도를 높일 수 있음
- 2번 시나리오는, 1번 시나리오에서 쌓은 이해도를 기반으로 하는 고도화 작업을 필요로 함

<br>

실제 SNS 서비스 사용자의 규모는 엄청나게 크기 때문에, 수행 시간은 중요한 성능 요소임

- 하지만, 언어와 컴퓨터 환경에 따라 차이가 생기므로, 직접적인 평가 요소로 반영하진 않음
- 즉, 시험 시간 내에 목표를 달성할 수 있는 로직을 구현하도록 하는 것이 문제의 의도였음

### 다양한 접근 방법

- 랜덤 추천 방식은 가장 간단한 방법이지만, 목표 달성까지 많은 수의 시뮬레이션이 필요함
- 팔로우 확률이 높은 순서대로 추천하는 그리디 방식은 정렬이 필요해서 수행 시간이 길어짐
- 그리디 방식에 휴리스틱 이론을 적용하여, 팔로우 확률이 아주 높은 사용자만 추천해도 됨
- 그리디, 휴리스틱 이론을 활용하면서 정렬을 생략하면, 횟수는 늘지만 수행 시간은 줄어듦
- 모든 사용자의 팔로우 순서를 동일하게 하여, 확률이 점차 늘어나도록 만드는 방법도 있음

## 8. 2021 KAKAO BLIND RECRUITMENT(1차)

> 출처 : ['2021 카카오 신입공채 1차 온라인 코딩 테스트 for Tech developers 문제해설'](https://tech.kakao.com/2021/01/25/2021-kakao-recruitment-round-1/)

- 모든 테스트 케이스를 통과해야만 정답 처리, 효율성 테스트가 있는 문제는 부분 점수 적용

### 8-1. 아이디 추천

> 입력받은 아이디를 주어진 조건에 맞춰 변경하기(문자열 처리)

> 정답률 : 57.50%

- 난이도가 가장 낮은 몸풀기 문제이며, 제시된 조건의 내용대로 정확하게 구현만 하면 됨
- 입력의 크기가 n일 때, 작성한 코드의 성능이 O(n^2) 이어도 제한 시간을 만족할 수 있음
- 조건에 맞는 문자만 추려서 새로운 문자열을 만들면, O(n) 성능의 코드를 작성할 수 있음

### 8-2. 메뉴 리뉴얼

> 주문 내역을 기준으로 코스 요리 메뉴 만들기

> 정답률 : 25.40%

- 주문 내역을 기준으로 가능한 모든 조합을 만든 후에, 가장 많이 생성된 조합을 찾으면 됨

### 8-3. 순위 검색

> 지원자 정보와 문의 사항을 기준으로, 문의 사항의 조건을 만족하는 지원자 수 구하기

> 정답률 : 정확성 44.07% / 효율성 4.49%

- 문의 사항마다 지원자 정보 목록을 확인하는 방식으로 정확성 테스트를 통과할 수 있음
- 하지만, 이렇게 매번 지원자를 확인하는 방법으로 풀면 효율성 테스트를 통과할 수 없음
- 효율성 테스트를 통과하려면, 지원자들을 조건 별로 미리 분류하여 따로 저장해두어야 함
- 이렇게 분류된 정보를 이용하면, 문의 사항이 될 수 있는 모든 경우의 수를 따질 수 있음
- 점수를 제외한 분류 기준이 총 4개이므로, 1명당 따져야 하는 조합의 수는 2^4 = 16개임
- 문의 사항마다 모든 지원자를 확인하지 않아도, 분류된 정보에서 원하는 값을 얻을 수 있음

### 8-4. 합승 택시 요금

> 그래프의 특정 위치에서 최소한의 비용으로 두 군데의 목적지를 모두 거쳐 가는 방법 찾기

> 정답률 : 정확성 9.60% / 효율성 7.45%

- 임의의 노드 i에서, 다른 임의의 노드 j로 이동하는 데에 필요한 최소 비용을 구해야 함
- '그래프에 있는 각 노드 사이의 최단 경로를 구하는 알고리즘' 을 활용하여 풀 수 있음
- '다익스트라(Dijkstra)', '플로이드-워셜(Floyd-Warshall)' 등의 알고리즘을 사용하면 됨
- '두 사람이 합승하지 않는 경우' 는 '합승 종료 지점이 시작점인 경우' 라고 할 수 있음
- 최단 경로 알고리즘을 이용해 각 지점 간의 비용을 계산한 후, 완전 탐색을 수행하면 됨

### 8-5. 광고 삽입

> 시청 내역을 기준으로, 가장 효율이 좋은 광고 시간대 찾아내기

> 정답률 : 1.23%

- '시간 정보를 초로 환산한다.' 라는 아이디어만 떠올린다면, 비교적 쉽게 풀 수 있는 문제임
- 영상의 최대 길이는 "99:59:59", 즉, 100시간이며, 이는 100 * 60 * 60 = 360,000 초임
- 최대 크기가 그렇게 크지 않기 때문에, 배열을 사용하여 초 단위로 시청자 수를 파악해도 됨
- 입력으로 주어지는 영상 길이, 광고 영상 길이, 시청 기록을 모두 초 단위로 환산할 수 있음
- 시청 기록의 시작/종료 시간을 이용하면, 전체 시간 배열에 초당 시청자 수를 합산할 수 있음
- 영상의 시작 부분부터 광고 삽입 시점을 1초씩 뒤로 늦추면서, 전체 시청자 수를 파악하면 됨

### 8-6. 카드 짝 맞추기

> 조작 횟수를 최소화하여 모든 카드의 짝 맞추기(경우의 수 완전 탐색, 너비 우선 탐색)

> 정답률 : 0.95%

- 카드의 종류가 최대 6개이므로, 선택 가능한 카드 짝 맞추기 순서의 개수는 6! = 120 개임
- 같은 종류의 카드 중, 어떤 카드를 먼저 제거하는지에 따라 조작 횟수가 달라질 수 있음
- 따라서, 짝 맞추기 순서와 카드 제거 순서를 모두 확인하면서, 최소 조작 횟수를 찾으면 됨

### 8-7. 매출 하락 최소화

> 매출 하락을 최소화하면서, 모든 팀에서 한 명 이상의 직원을 워크숍에 참가시키는 방법 찾기

> 정답률 : 0.57%

- 트리 자료 구조를 잘 이해하고 있고, 다이나믹 프로그래밍을 활용할 수 있어야 풀 수 있음
- 특정 노드의 최적해를 구하려면, 우선, 해당 노드의 서브 트리에 대한 최적해를 구해야 함
- 따라서, 루트 노드로부터 시작하여 깊이 우선 탐색을 수행하면서 각각의 최적해를 구하면 됨
- 팀장이 워크숍에 참가하지 않는 경우, 팀원 중 한 명을 무조건 워크숍에 참가시켜야 함
- 팀원이 참가하는 경우, 해당 팀원의 참가 여부에 따른 매출 손해의 차이를 최소화해야 함

## 9. 2021 KAKAO BLIND RECRUITMENT(2차)

> 출처 : ['2021 카카오 신입공채 2차 온라인 코딩 테스트 for Tech developers 문제해설'](https://tech.kakao.com/2021/02/16/2021-kakao-recruitment-round-2/)

- 코로나19 바이러스로 인해, 오프라인으로 진행되던 테스트가 온라인(비대면) 테스트로 변경됨

### CS 테스트

- 단답형 주관식 1문제와 객관식 9문제가 출제되었으며, 총 15분의 풀이 시간이 제공됨
- 자료 구조(3) + 데이터베이스(2) + 네트워크(2) + 운영 체제(2) + 알고리즘(1) = 시험(10)
- 지원자 대부분이 어렵지 않게 통과할 수 있는, CS 기초 지식을 검증하는 수준으로 출제됨

### 코딩 테스트

- CS 테스트 이후 15분의 쉬는 시간이 주어졌으며, 코딩 테스트는 4시간 45분 동안 진행됨
- 2차 코딩 테스트는 1차 코딩 테스트와는 달리, 정답 맞히기가 아닌 점수를 높이는 방식임
- 기본기와 경험을 바탕으로 요구 사항을 이해하고, 로직을 설계/개선하는 역량이 필요함
- REST API와 JSON을 활용하는, 문제 지문 이해와 풀이법 도출에 시간이 꽤 필요한 문제임
- 따라서, 지원자는 API 호출을 수행하는 모듈과 JSON 파서를 미리 준비해둘 필요가 있음

### 카카오 T 바이크 관리 시뮬레이션

> 문제 내용 : 지역마다 대여 수요가 다를 때, 전기 자전거가 부족하지 않도록 재배치하기

> 목표 : 최소한의 트럭 운영을 통해, 전기 자전거가 부족한 지점의 수를 최소화해야 함

### 서비스 지역 및 제약 사항 설명

- 자전거 대여/반납이 가능한 서비스 지역은, n * n 크기의 정방형 격자 모양으로 주어짐
- 격자의 한 칸을 구성하는 한 변의 길이는 100m이며, 한 칸마다 대여소가 하나씩 있음
- 대여소마다 고유한 ID 값을 가지며, 보관할 수 있는 자전거의 수는 무한대라고 가정함
- 사용자는 트럭이 운영하는 시간 동안만 요청할 수 있으며, 요청은 분 단위로 처리됨
- 사용자 요청은 대여할 대여소와 반납할 대여소의 ID, 자전거를 탈 시간(분) 으로 구성됨
- 사용자가 대여를 요청했을 때, 대여소에 자전거가 없다면 해당 요청은 자동으로 취소됨
- 트럭은 상하좌우로 1칸씩만 이동할 수 있으며, 지정된 서비스 지역을 벗어날 수 없음
- 트럭은 오전 10시부터 오후 10시까지 운행하며, ID 값이 0인 대여소에서 운행을 시작함
- 트럭은 1칸 이동, 싣기, 내리기에 6초가 걸리며, 최대 20대의 자전거를 수용할 수 있음
- 서버는 매분, 반납 처리 - 대여 요청 처리 - 트럭 운행 처리의 순서대로 작업을 처리함

### 시나리오

> 시나리오 1 : 서비스 지역은 5 * 5, 평균 분당 요청 2건, 자전거는 100대, 트럭은 5대

> 시나리오 2 : 서비스 지역은 60 * 60, 평균 분당 요청 15건, 자전거는 10,800대, 트럭은 10대

- 시나리오 2에는 대여 요청이 가장 많이 일어나는 선호 대여소(HotPlace) 가 1곳 지정됨
- 선호 패턴은 4시간마다 변경되므로, 트럭 운영 시간 동안 서로 다른 3개의 패턴이 등장함
- 선호도가 가장 높은 대여소와 가장 낮은 대여소의 요청/반납 건수의 차이는 약 60배임
- 선호도는 최근 3일간의 대여 요청이 분별로 저장된 과거 요청 기록을 기준으로 결정됨

### 점수

- 점수는 시나리오별로 계산되며, 달성률 점수, 효율성 점수, 총점으로 구분되어 계산됨
- 작성한 코드는 시간 내에 업로드해야 하며, 각 시나리오의 총점은 최고 점수로 반영됨
- 시나리오마다 배점 가중치가 다르게 적용되며, 최종 점수는 (총점) * (가중치) 로 부여됨

### API REFERENCE, Response Codes

- HTTPS를 이용해 통신하며, 초당 10회 이상의 API 호출은 서버가 응답하지 않을 수 있음
- 응답 코드는 API 호출 성공(200), 입력 문제(400), 인증 문제(401), 서버 에러(500) 임

### API

- Start API : 문제 풀이에 필요한 key 발급, 파라미터에 맞춰 대여소/트럭 정보 초기화
- Locations API : 현재 시점에서 각 자전거 대여소가 보유한 자전거의 수 반환
- Trucks API : 현재 시점에서 각 트럭의 위치와 해당 트럭이 싣고 있는 자전거의 수 반환
- Simulate API : 현재 시점부터 1분 동안 각 트럭을 어떻게 운영할지를 서버에 전달
- Score API : Start API로 발급받은 key를 입력받아, 해당 key로 획득한 점수 반환

### 문제 접근 방법

- 시나리오는 난이도를 기준으로, 점수는 달성률과 효율성으로 구분된다는 것을 파악해야 함
- 또한, 점수를 높이려면, 성공 건수를 늘리고, 트럭 이동을 줄여야 한다는 것도 파악해야 함
- 이 때, 효율성보다 달성률의 점수 가중치가 높아서, 성공 건수를 늘리는 것이 더 효율적임
- 그리고, 트럭 운행을 하지 않았을 때보다 성공 건수가 많아야 달성률 점수를 획득할 수 있음
- 따라서, Simulate API를 활용해 달성률을 점검하면서 로직을 개선하는 식으로 진행해야 함

1. 아무것도 하지 않기
2. 자전거를 가장 많이 보관 중인 대여소에서 가장 적게 보관 중인 대여소로 옮기기
3. 자전거 수가 가장 적은 대여소 a에 대해, a 주변에 있는 대여소의 자전거 1대를 a로 옮기기
4. 3번 방법을 그대로 처리하되, 1대의 자전거가 아닌 여러 대의 자전거를 옮기기
5. 요청이 발생했을 때, 대여 장소에 자전거 1대를 옮기고, 반납 장소로 트럭 옮기기
6. 트럭마다 구역을 정하고, 자전거가 부족한 대여소에 남는 자전거를 옮기도록 만들기
